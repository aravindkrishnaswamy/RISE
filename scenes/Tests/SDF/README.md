# SDF (Signed-Distance-Field) Geometry Tests

`sdf_geometry` is RISE's first *implicit* primitive: a list of transformed
distance primitives composed with hard- or **smooth**-boolean operators and
ray-traced by sphere tracing. It exists for melded / filleted organic shapes
that the analytic primitives and hard-boolean CSG cannot express (a watch lug
flowing into a bezel with a real fillet, a fat-then-tapering hand, a metaball
blob). Implementation: `src/Library/Geometry/SDFGeometry.{h,cpp}`.

## The `sdf_geometry` scene chunk

Parts are authored **inline** with repeatable `part` lines — the SDF lives in
the scene file itself, no sidecar files:

```
sdf_geometry
{
	name			blobgeom
	part			roundbox union 0  0 0 2  0 0 0  1 1 1  2 2 2  0.5
	part			sphere smin 1.6  2.6 3.2 0  0 0 0  1 1 1  1.7 0 0  0
	maxsteps		256       # optional: sphere-trace step cap (default 256)
	epsilon			5e-5      # optional: surface eps as a fraction of the bbox diagonal
	sampling_detail	64        # optional: tessellation cells (longest axis) for area-light / SSS sampling
}
```

Parts compose **in order** (each line folds into the running field with its
op).  For a very large SDF an external file is still supported — `file
path/to/parts.txt`, same one-part-per-line grammar (no count header; blank
lines and `#` comments allowed) — but provide either `part` lines or `file`,
never both (hard parse error).

The geometry is **object-space**: position / orient / scale it with the
enclosing `standard_object` exactly like any other geometry. It participates
in the normal ray pipeline — primary rays, shadow rays, reflection/refraction
continuation rays, and the caustic photon pass all sphere-trace against it.

## The part-line grammar

One grammar for inline `part` lines and external parts files — both are
parsed by `SDFGeometry::ParsePartLines`, so they cannot drift.  Future
shapes / ops / per-part fields extend that one parser; the scene chunk
forwards `part` lines verbatim and needs no change.

```
<type> <op> <k>  <px py pz>  <exDeg eyDeg ezDeg>  <sx sy sz>  <a b c>  <round>
```

- **type**: `sphere` `box` `roundbox` `cylinder` `torus` `capsule` `roundcone`
- **op**:   `union` (min) · `smin` (smooth union, `k` = blend radius) · `subtract` · `intersect`.
  The **first** part must be `union` or `smin` — the field starts empty, so a
  leading `subtract`/`intersect` would compose against nothing and yield an
  always-miss SDF; the parser hard-fails it.
- **k**: smooth-min blend radius in object units (0 = hard)
- **pos / euler(Rz·Ry·Rx, degrees) / scale**: per-part transform (non-uniform scale OK)
- **a,b,c**: primitive size — sphere `a`=radius · box `a,b,c`=half-extents · cylinder `a`=radius `b`=half-height (axis = local Y) · torus `a`=major `b`=tube (ring in XZ) · capsule `a`=radius `b`=half-height · roundcone `a`=base-radius `b`=tip-radius `c`=height (axis = local Y, base at y=0)
- **round**: extra corner rounding (roundbox)

The watch's chunks are generated by
`scenes/FeatureBased/GuillocheWatch/sdf_gen.py` (prints inline `sdf_geometry`
chunks ready to paste into `watch_dial.RISEscene`).

## Unit test

`tests/SDFGeometryTest.cpp` (126 checks) validates the SDF against closed forms:
SDF sphere ≡ analytic sphere (range + normal over many directions), box face
distances, `smin` bulges the neck, bbox containment, inside-start exit, rigid
transform, non-uniform scale, plus the boolean ops: **subtract** (hard *and*
smooth k=1) carves a real cavity without tunnelling a wall or filling the hole,
**intersect** produces a lens with exact hit positions, and
`IntersectRay_IntersectionOnly` agrees with `IntersectRay` on the front/back
face filter, plus the first-class surface machinery: GetArea vs closed
forms, on-surface/uniform sampling, TessellateToMesh contract (indices, areas,
outward winding, append offsets), and a 400k-sample MC integration of the NEE
integrand against the analytic answer (ratio 1.000).  Test 20 covers the
shared part-line grammar (`ParsePartLines`): field-exact parsing with
comments / blank lines, hard failures on unknown / malformed / trailing
tokens and the retired count-header format, and hit-identity between parsed
and directly-constructed parts.  Test 21 pins the first-op rule (leading
`subtract`/`intersect` rejected; leading `smin` hits identically to `union`).
Test 22 pins the missed-component detector (sub-cell nub: rendered but
unsampled + flagged + `CanBeAreaLight()` withdrawn at detail 8; sampled +
unflagged + capability restored at detail 64).  Test 23
pins the curvature-corrected `GetArea` against closed forms.

## Why subtract / intersect don't overstep

Subtract / intersect compose with the polynomial smooth-max, whose *value*
exceeds the hard max — which looks like a sphere-trace overstep hazard. It isn't:
`smin`/`smax` are **1-Lipschitz** given 1-Lipschitz inputs (the blend gradient is
a convex combination of two unit gradients), and each part field is ≤ 1-Lipschitz
(`minScale` compensates non-uniform scale). So the composed `Map` is ≤ 1-Lipschitz
and vanishes on its own zero set → `|Map(p)| ≤ true distance`, a conservative
sphere-trace bound for *any* blend radius. Test 10/11/12 are the empirical proof.

## Bad part tokens hard-fail the load

An unknown primitive or op token (a typo like `round_box` or `substract`), a
malformed line, or trailing extra tokens abort the parse with
`source / line / token` context — the source being the scene's inline part
list or the parts-file path — rather than silently falling back to
`sphere` / `union` and rendering a different object.  Providing both `part`
lines and a `file` (or neither) is likewise a hard error, diagnosed with the
chunk's `name`.

## SDF is a first-class surface: area lights, SSS, and mesh generation

The SDF lazily tessellates itself (marching **tetrahedra** over the padded bbox —
Freudenthal 6-tet split, face-consistent, no ambiguous cases — with every vertex
Newton-projected onto the zero set) and builds a Morton-ordered triangle CDF at
`sampling_detail` (chunk param, default 64), with each triangle weighted by
chord area × the **projection Jacobian** (J ≈ 1 − d·(κ₁+κ₂), 3-point
edge-midpoint quadrature): sampling draws a point on a planar chord triangle and
projects it onto the curved zero set, and that projection is not
area-preserving, so the raw chord measure would make the realized density
disagree with the `pdfPosition = 1/GetArea()` the light sampler assumes by
O((cell/curvature radius)²).  The J-weighted CDF and J-corrected area cancel
the first-order term; what remains is the zero-mean within-triangle Jacobian
residual and the J clamp ([0.5, 2]) near creases.  That backs:

- **`GetArea()`** — the curvature-corrected sampling area (sphere r=1.5 at
  detail 64 within 0.007% of 4πr², torus 1.45/0.55 within 0.05% of 4π²Rr — vs
  ~1.3% raw chord deficit on the torus; a carved box-with-tunnel lands within
  ~1.4% of the closed form — marching methods chamfer sharp edges slightly).
  This is the area of the surface the **sampler can see**: components the
  sampling grid missed entirely (below) are not in it.
- **`UniformRandomPoint()`** — J-weighted CDF triangle pick (with the pick
  variable's remainder reused as a fresh uniform, so only two sampler dims are
  consumed, matching every other geometry), √-warp barycentric, projection onto
  the sphere-traced surface, exact gradient normal. The offline closed-form
  harness (`SDFGeometryTest` Test 19) pins the NEE estimator at **ratio 1.000**.
- **Missed-component detector** — marching tets only see sign changes at cell
  corners, so a feature thinner than a sampling cell (a tiny nub, a thin tube)
  can render via sphere tracing yet be absent from the CDF: as an emitter it
  would never be NEE-sampled while BSDF-hit MIS still budgets a light-sampling
  density for it.  `EnsureSamplingStructure` probes every no-sign-change cell's
  CENTER; a disagreeing center is a *provable* miss —
  `SuspectedMissedFeatureCells()` reports the count and a build-time warning
  names the first location and tells you to raise `sampling_detail`.  This is a
  best-effort lower bound (a sub-half-cell feature can evade the center probe
  too); the guarantee is the test pair in `SDFGeometryTest` Test 22 (nub
  missed + flagged at detail 8, captured + quiet at detail 64).  A proven miss
  **withdraws the capability**: `CanBeAreaLight()` returns false, so the SDF is
  refused as a luminary / SSS source and BSDF-hit emission takes full MIS
  weight — unbiased but noisy — instead of silently re-introducing the
  phantom-pdf bias on exactly the missed surface.  All three integrators
  honour the withdrawal: PT and the emission shaderops gate on
  `CanBeAreaLight()` directly, BDPT zeroes `pdfPosition` for zero-area
  emitters, and VCM's s=0 strategy zeroes the competing light pdf the same
  way (it previously **dropped** the emission of a zero-sampling-area SDF —
  black emitter under VCM only).  SMS surface seeding shares the same
  contract: an SDF that actually classifies as a specular caster but has
  `CanBeAreaLight() == false` is refused as a seeding caster (warning; its
  caustics are not SMS-seeded) — the gate sits after specularity
  classification, so diffuse/emissive SDFs neither warn nor consult the
  contract — rather than
  seeding from a partial surface or the degenerate empty-mesh fallback
  point — `sms_seeding "snell"`'s deterministic trace is unaffected except
  for its pure-mirror supplemental loop, which reads the same caster list.
- **Area lights**: an emissive SDF registers as a real luminary and is
  NEE-importance-sampled — see `sdf_arealight.RISEscene` (a glowing SDF torus
  lighting a floor under pure DirectLighting). Linear-HDR floor brightness
  matches an analytic emitter of equal area within MC noise. *Caveat*: an SDF
  emitter measures ~2× the pixel variance of an analytic-primitive emitter at
  equal samples (a surface-curve CDF map fragments sampler stratification at the
  visibility boundary, where a smooth parametric map does not) — correct, just
  noisier; prefer analytic emitters when the shape allows. Beware judging A/Bs
  through 8-bit sRGB PNGs: clipping + the concave transfer curve convert that
  variance gap into a fake "brightness deficit" (~4% here); compare linear HDR.
- **Subsurface scattering**: the point-set SSS shaderops build their irradiance
  caches from the same sampler — SDF + `simple_sss_shaderop` works.
- **`TessellateToMesh`** — the same mesher serves the public tessellation API
  (`detail` = cells along the longest bbox axis, clamped 8..512), so
  `displaced_geometry` can wrap an SDF (displacement-mapped SDFs) and meshes can
  be exported. Triangles are wound outward; normals are exact field gradients.

## Cross-integrator validation (PT / BDPT / VCM)

The SDF luminaire was A/B'd against analytic emitters under all three
integrators (linear-HDR patch means; PT as reference):

- **BDPT, vacuum** (`sdf_arealight` + `bdpt_pel_rasterizer`): floor ratio
  **0.977** — light subpaths start on the SDF via the same
  `UniformRandomPoint`/`pdfPosition = 1/area` machinery, MIS clean, no
  splotches.
- **BDPT, fog** (`sdf_luminaire_fog`): floor ratio **1.003**. The volumetric
  in-scatter patches read +18–27% over PT — and the analytic-sphere control
  reads **+29–37%** on the same patches, so that delta is a pre-existing
  BDPT-vs-PT volume characteristic (or slow BDPT volume convergence), not the
  SDF: the SDF tracks PT *better* than the analytic control does.
- **VCM**: floor ratio **0.988** (was ~0.36× before 2026-06-10 — that deficit
  was two stacked non-MIS bugs, both fixed: the OIDN-on plain file output
  dropped the t=1 splat film, and the analytic torus's `UniformRandomPoint`
  rejection-retry had a point-mass; see
  `docs/skills/bdpt-vcm-mis-balance.md` step 0.  The analytic-torus control now
  reads 1.0007.)  VCM's volumetric in-scatter deficit matches the documented
  RISE-wide VCM volume luminance bias (see
  `docs/UNIFIED_INTEGRATOR_DECISION.md`: VCM is the right integrator for
  caustic transport only).

The property this geometry is responsible for — *behaving indistinguishably
from analytic geometry under every integrator* — holds in all measurements.

`CanBeAreaLight()` reports honestly: `GetArea() > 0 &&
SuspectedMissedFeatureCells() == 0` — false for a field that tessellates to
nothing **and** for one where the detector proved the sampling mesh missed
renderable surface (the contract "UniformRandomPoint covers what renders" is
what every consumer relies on).  The guards in `LuminaryManager`, the emission
MIS sites, and the SSS shaderops all key off it, so a refused SDF degrades
consistently everywhere: no NEE registration, emission at full BSDF-hit weight,
no SSS — unbiased, just noisier, with the build-time warning naming the missed
location and the `sampling_detail` cure.

The sphere-trace marcher also refines every reported hit to the **true zero
crossing** (bracket + bisect after band entry; a crossing beyond the allowed
range is *no hit*, and tangent band-passes resume the march instead of
manufacturing a hit) — so shadow rays toward light samples on the SDF's own
surface do not falsely self-occlude at grazing incidence, and phantom
grazing hits cannot occur.

## Constructing an SDF from the C API

`sdf_geometry` in a scene file and `IJob::AddSDFGeometry` both route through the
C-API construction boundary `RISE_API_CreateSDFGeometry(ppi, filename, parts,
maxSteps, epsilon, samplingDetail)` — pass the part lines as a newline-separated
string in `parts` (the normal path) or a parts-file path in `filename`, exactly
one of the two.  The grammar itself lives in `SDFGeometry::ParsePartLines`.
External C clients construct an SDF the same way every other geometry is
built — via the `RISE_API_Create*Geometry` factory — rather than only through
scene files.

## Authoring note — continuation rays start *on* the surface

`PerfectRefractorSPF` (and the reflection path) spawn the next ray with its
origin exactly at the hit point, no offset. Sphere tracing must therefore step
off the originating surface band before choosing its march direction, or a
refracted ray instantly re-hits the surface it came from (symptom: **black
glass**). `SDFGeometry::March` does this — it is the SDF analogue of how the
analytic primitives reject the `t≈0` self-intersection root. If you add a new
SDF primitive or op, keep that escape intact.
