## Quickstart

- Linux/macOS main build: `make -C build/make/rise -j8 all`
- Linux/macOS tests: `make -C build/make/rise tests` then `./run_all_tests.sh`
- Windows tests: `.\run_all_tests.ps1`
- Windows tests (Debug): `.\run_all_tests.ps1 -Config Debug`
- Sample render:

```sh
export RISE_MEDIA_PATH="$(pwd)/"
printf "render\nquit\n" | ./bin/rise scenes/Tests/Geometry/shapes.RISEscene
```

### One-time setup: OIDN with GPU support (extlib submodule)

OIDN is shipped as a git submodule at `extlib/oidn/source` so RISE
links against an in-tree install with GPU device backends enabled
(Metal on macOS; CUDA / HIP / SYCL toggles on Windows). System OIDN
installs from Homebrew, vcpkg, etc., are typically built with CPU
device only ‚Äî RISE's denoise then silently runs on CPU even on a
Metal-capable Mac. Full background in [docs/OIDN.md](docs/OIDN.md)
(OIDN-P0-3 install gotcha).

Run once after cloning, and again whenever the OIDN submodule
commit is bumped:

```sh
# macOS / Linux
git submodule update --init extlib/oidn/source
extlib/oidn/build.sh
```

```powershell
# Windows (PowerShell, in repo root)
git submodule update --init extlib/oidn/source
pwsh -File extlib\oidn\build.ps1     # add -EnableCuda / -EnableHip / -EnableSycl as needed
```

Both scripts populate `extlib/oidn/install/`; RISE's Makefile and
VS2022 project files prefer that path over any system OIDN. Verify
the selected device on your first render ‚Äî look for:
`OIDN: creating Metal device (one-time per rasterizer)` in the log.

Skipping this step is fine ‚Äî RISE falls back to the system OIDN
(typically Homebrew's CPU-only build) and denoise still works,
just without GPU acceleration.

## Source-file add/remove ‚Äî touch ALL five build projects

When you add **or remove** any `.cpp` / `.h` under `src/Library/`, update every one of these; none auto-discovers files:

1. `build/make/rise/Filelist` (Unix/Linux, canonical)
2. `build/cmake/rise-android/rise_sources.cmake` (Android NDK, consumed by `build/cmake/rise-android/CMakeLists.txt` which the Gradle build invokes)
3. `build/VS2022/Library/Library.vcxproj` (Windows, tracked in git) ‚Äî `<ClCompile>` for `.cpp`, `<ClInclude>` for `.h`.
4. `build/VS2022/Library/Library.vcxproj.filters` ‚Äî same entries with `<Filter>` tags so the Solution Explorer tree matches the build.
5. `build/XCode/rise/rise.xcodeproj/project.pbxproj` ‚Äî Xcode references appear in four sections (`PBXBuildFile`, `PBXFileReference`, the containing `PBXGroup`, and per-target `Sources` / `Headers` build phases) and usually in two targets (library + GUI); delete every match when removing a file.

Full rules, exclusions, and related update surfaces (API, parser chunks, tests, scenes) are in [AGENTS.md](AGENTS.md) ‚Üí "Change Checklist".

## Compiler warnings are bugs

Don't introduce warnings.  Don't leave them.  Don't suppress them with
pragmas or `-Wno-*` flags ‚Äî fix the root cause.  Both the `make` build
and the Xcode `RISE-GUI` build must come up warning-free on a *clean*
rebuild before you call a task done ‚Äî incremental builds hide warnings
on files that didn't recompile.  Full rationale, the principled fix per
warning kind, and the verification commands are in [AGENTS.md](AGENTS.md) ‚Üí
"Compiler Warnings Are Bugs".

## High-Value Facts

- **`IScalarPainter` is the physical-scalar pipe; `IPainter` is the colorspace+JH-uplift pipe ‚Äî material slots route by physical meaning, not by syntactic resemblance** ([docs/ISCALARPAINTER_REFACTOR.md](docs/ISCALARPAINTER_REFACTOR.md)).  Before 2026-05 every material parameter went through `IPainter` regardless of whether it was a *color* (reflectance, emission, tau-as-tint) or a *physical scalar* (IOR, roughness, scattering coefficient, absorption, phase-function asymmetry).  The spectral renderers' `GetColorNM` path JH-uplifts inline-numeric RGB values via the Jakob‚ÄìHanika spectral coefficient LUT ‚Äî which silently mangled values like `scattering 1000000` (clamped to ~1.0 in the visible band) and rendered glass spheres invisible in every spectral rasterizer.  The refactor (Phases 1‚Äì9) introduced a parallel `IScalarPainter` interface that **never goes through JH uplift**.  Member types: a slot's "color-ness" or "scalar-ness" is encoded in its TYPE ‚Äî `const IPainter&` vs `const IScalarPainter&`.  Every material's SPF / BSDF / Material header has been audited (`docs/ISCALARPAINTER_REFACTOR.md` Phase 4 list).  When adding a new material parameter, ask "is this a tinted attenuation/reflectance/emission (per-wavelength sample)?" ‚Üí `IPainter`.  "Is this a physical-scalar coefficient (per-wavelength magnitude, no colorspace conversion)?" ‚Üí `IScalarPainter`.  Pre-refactor scenes that bound `uniformcolor_painter` / `spectral_painter` chunks to scalar slots are auto-migrated by `tools/migrate_scenes_iscalarpainter.py`; remaining `spectral_painter` references need a hand-edit (replace with `scalar_painter { file ... }`).  At parse time, `Job::Add*Material` emits a per-parameter diagnostic distinguishing "name is bound to an IPainter chunk" vs "name is unknown" ‚Äî surface that diagnostic to authors instead of guessing.  Transitional `PainterToScalarAdapter` (`src/Library/Painters/PainterToScalarAdapter.h`) bridges existing `BlendPainter` chains (PBRMetallicRoughness anisotropy graph) into IScalarPainter slots ‚Äî fine for spatially-varying grayscale inputs, **does not protect against JH uplift on a spectral input**.  **Don't add a `GetColorNM` override to `IScalarPainter`** to "fix" anything ‚Äî the entire point of the interface is the absence of that path.

- **MIS heuristic per integrator ‚Äî BDPT power-2 / VCM balance is intentional, not a mismatch** ([docs/MIS_HEURISTICS.md](docs/MIS_HEURISTICS.md)): RISE PT (BSDF vs NEE) uses power-2 with optional optimal-MIS (Kondapaneni 2019); BDPT uses power-2 ([BDPTIntegrator.cpp:4378-4575](src/Library/Shaders/BDPTIntegrator.cpp), `MISWeight` walk, `ri * ri` accumulation; compile-time `MISWEIGHT_BALANCE_HEURISTIC` toggle exists for diagnostic A/B but is off in production); MLT inherits BDPT; VCM uses balance ([VCMRecurrence.h:56](src/Library/Shaders/VCMRecurrence.h:56), `VCMMis(x) = x`); volumetric DT-vs-equiangular uses 50/50 balance.  **The asymmetry matches PBRT-v4, Mitsuba 0.6/3, SmallVCM, and SmallUPBP** ‚Äî it is mandated by the Georgiev 2012 dVCM/dVC/dVM running-quantities recurrence, which only closes algebraically at Œ≤=1.  Switching VCM to power-2 would either break the O(1)-per-vertex MIS evaluation (since `(Œ£ p_i)¬≤ ‚âÝ Œ£ p_i¬≤` and tracking sum-of-squares of partial pdf products requires a separate running quantity per strategy) or collapse VCM's merge path back to BDPT-style explicit enumeration, defeating its core architectural advantage.  A second reason: once SPPM-style auto-radius shrinks per-sample, merging vs connection PDFs differ by ~10 orders of magnitude ‚Äî squaring those ratios pushes them into FP under/overflow and forces winner-takes-all between two strategies that both legitimately contribute in caustic regions.  **Don't propose "fixing" this asymmetry**; the principled forward direction is extending optimal-MIS (already in PT) to BDPT and VCM, not flipping the heuristic.  Tungsten BDPT uses balance and is the documented outlier (~10-30 % variance left on the table).  When BDPT and VCM disagree on caustic-heavy scenes it is a transport-coverage difference (BDPT can't reach S-D-S-E without merging), not an MIS-weighting issue ‚Äî walk through [docs/skills/bdpt-vcm-mis-balance.md](docs/skills/bdpt-vcm-mis-balance.md) before suspecting the heuristic.

- **SMS seeding mode** (`sms_seeding` parameter, [docs/SMS_UNIFORM_SEEDING_PLAN.md](docs/SMS_UNIFORM_SEEDING_PLAN.md), measurements + regime analysis in [docs/SMS_UNIFORM_SEEDING_RESULTS.md](docs/SMS_UNIFORM_SEEDING_RESULTS.md), background in [docs/SMS_FUTURE_DIRECTIONS.md](docs/SMS_FUTURE_DIRECTIONS.md)): two strategies ‚Äî `"snell"` (default; deterministic Snell-traced refraction seed from shading point toward sampled light + a pure-mirror caster supplemental loop that uniform-area-samples each pure-mirror caster `M = multi_trials` times to recover diacaustic-style chains) and `"uniform"` (Mitsuba-faithful uniform-area sample on cached caustic-caster shapes per `manifold_ss.cpp` / `manifold_ms.cpp`).  Path-tree branching at sub-critical dielectric Fresnel splits was excised in 2026-05 (matches Mitsuba SOTA); reflection caustics through dielectric chains that previously relied on it are now out of scope for SMS ‚Äî use VCM or photon mapping instead.  **The two seeding modes are regime-complementary, not one-better:**
  - **Use `"snell"` (default) for displaced/bumpy specular casters and refraction-dominated caustics.**  The deterministic Snell-trace seed is by-construction in the dominant refraction-caustic basin; Newton converges from there even when per-bump basins are too small for uniform-area sampling to reliably hit.  Measured: displaced Veach egg snell ratio 0.37 vs uniform 0.008 ‚Äî uniform-area collapses on heavy displacement.
  - **Use `"uniform"` for smooth analytic primitives with reflection/glint caustics** (mirror chains, diacaustics, lensed reflections off smooth dielectrics).  Uniform-area covers the full caster surface so reflection-root basins on smooth geometry get hit; Snell-trace toward the light can't seed those chains.
  - On scenes that match neither regime cleanly, both modes give similar ratios within MC noise (smooth Veach 0.93 vs 0.94, k1_botonly tied).  Default `"snell"` is the safer choice when in doubt.

  Photon-aided extension (Weisstein 2024 PMS) layers on top of biased uniform mode and the snell-mode multi-trial loop alike.  Required `seedingMode = Uniform` for principled geometric Bernoulli `1/p` (Zeltner 2020 ¬ß4.3) when `sms_biased FALSE`.

- **SMS two-stage solver scope** (`sms_two_stage` parameter, [docs/SMS_TWO_STAGE_SOLVER.md](docs/SMS_TWO_STAGE_SOLVER.md)): the Zeltner 2020 ¬ß5 two-stage solver helps on **smooth analytic primitives with normal-perturbing maps** (verified: bumpmap on `ellipsoid_geometry` reaches Œ£L_sms/Œ£L_supp = 1.15 at gentle bumps).  It **regresses** on **heavily-displaced meshes** (`displaced_geometry` with `disp_scale` > ~10 % of curvature radius): Stage 1 converges to the smooth-surface caustic, whose corresponding mesh position is up to `disp_scale` units away from the bumpy caustic ‚Äî Stage 2's seed is *worse* than the original Snell-traced on-mesh seed.  This matches Mitsuba's reference scope: their [`Figure_9_Twostage`](https://github.com/tizian/specular-manifold-sampling/tree/master/results/Figure_9_Twostage) scenes use only smooth primitives + `normalmap` BSDFs; their displaced-mesh comparison ([`Figure_16`](https://github.com/tizian/specular-manifold-sampling/tree/master/results/Figure_16_Displacement)) never engages two-stage.  Mitsuba's smoothing is plumbed via `bsdf->lean()` (LEAN moments, Olano-Baker 2010) ‚Äî a no-op for non-normal-mapped BSDFs.  **Default is OFF; turn ON only for the smooth-primitive + normal-map regime.**  For heavily displaced caustics, prefer VCM.

- **SMS dedupe + sum-clamp parity across {snell, uniform} √ó {RGB, NM} (2026-05)**: cross-mode audit (commits `e853ab8`, `bb166f5`, `3e1fe32`) brought every SMS code path's two safeguards in line.  Both were drifting: (a) **Dedupe key**: only RGB snell mode tracked Fresnel-branch siblings via a per-vertex-isReflection bitmask in addition to (firstPos, chainLen).  RGB uniform, NM uniform, and NM snell all used position-only or position+chainLen, silently collapsing chains that share a first vertex but flip reflect/refract at a later vertex (e.g. one chain refracts at v1 and a sibling reflects).  Now all four use `(firstPos, chainLen, reflectMask)` via a local `buildReflectMask` lambda.  (b) **`maxGeometricTerm` clamp**: only RGB snell mode applied the cap to the SUM of geometric terms across all unique preimages; the other three applied it per-path inside `ComputeTrialContribution{,NM}`.  N-preimage fold caustics could emit up to `N √ó maxGeometricTerm` instead of bounded total.  Fixed by adding `bool clampGeometric = true` + `Scalar* outSmsGeometric = nullptr` parameters to the helpers (default preserves existing callers), and refactoring uniform mode + NM snell to store unclamped per-trial pre-geo + smsGeometric, then post-loop sum-clamp matching the RGB-snell pattern at line ~7050.  Default behaviour unchanged on scenes that don't trip the bugs.  86/86 unit tests pass; canonical SMS scenes (sms_k1_refract, k2_glassblock, k2_glasssphere, k1_botonly) production-clean.

- **SMS unbiased-mode photon asymmetry is intentional (2026-05)** ([docs/SMS_PHOTON_REGIME.md](docs/SMS_PHOTON_REGIME.md) ¬ß"Mode/photon asymmetry in unbiased mode"): in `sms_biased FALSE`, snell mode uses photons but uniform mode skips them.  The asymmetry reflects different estimators: uniform's strict Mitsuba Bernoulli K-loop requires `proposal-matching` (every retry must use the same uniform-on-shape proposal as the main solve) ‚Äî mixing photons (which have a different proposal: light-emit-trace + kd-tree query) would bias the K count.  Snell uses `EstimatePDF`'s tangent-perturbation basin-width heuristic which is proposal-agnostic ‚Äî it perturbs around whatever seed it's given, so photons slot in cleanly.  Practical guidance: photons + unbiased = use snell; photons + uniform = use biased.  Documented in code at `EvaluateAtShadingPointUniform{,NM}` unbiased branch.

- **SMS photons help only when seeds miss basins (2026-05, displaced-egg empirical)**: re-ran the v7 displaced-painter sweep with `sms_photon_count 100000` enabled in BOTH snell and uniform modes (commit a3790c4 + 9bd7c5b photon-aware target-bounces filter).  Ran on the 66-cell painter √ó disp grid with `sms_target_bounces 2`.  Aggregate ratio change vs no-photons baseline:  **snell mean ratio 0.893 ‚Üí 0.910 (+1.9 %, statistical noise); uniform 0.351 ‚Üí 0.332 (‚àí5.5 %).**  Per-scene `sms_valid` count is ESSENTIALLY UNCHANGED (snell+t2: 66,402 valid contributions without photons vs 66,356 with ‚Äî i.e., photons converge to basins that snell-trace already covered, the basin-dedupe drops them, and net unique caustic coverage is the same).  Wall time: ~6√ó slower with photons.  Take-away: **photons help when the seed-builder MISSES BASINS** (e.g., diacaustic-class caustics where the deterministic Snell-trace can't follow reflection geometry, or scenes with multi-mirror chains).  On scenes where snell-trace already finds the natural caustic basin (the displaced-Veach-egg with interior dielectric shell, k=2 single basin), photons add cost without adding coverage.  Per-scene rule: only enable `sms_photon_count > 0` when there's evidence the snell-trace systematically misses caustic regions.  The default 0 stays the right answer for typical refraction-dominated scenes.  Note: this is an EMPIRICAL claim about photon utility on a single scene class; multi-basin / mirror-chain scenes (diacaustic, sms_slab_close_sms) may show different photon ROI ‚Äî see `docs/SMS_PHOTON_REGIME.md` for the broader characterisation.

- **SMS `sms_target_bounces` (2026-05, Mitsuba `m_config.bounces` analogue)**: per-rasterizer config wiring in BOTH snell and uniform modes that REQUIRES the seed chain to have exactly K specular vertices.  When `sms_target_bounces > 0`: (i) `SnellContinueChain` breaks at `chain.size() >= K`; (ii) `BuildSeedChainBranching` emits each branch as terminal at K; (iii) `BuildSeedChain` post-checks final length and rejects if `!= K`.  Default 0 = "no target": snell uses the emitter-projection cap (which discovers natural K), uniform uses `maxChainDepth` only.  **Set to the natural caustic K for the scene (typically 2 for a glass shell or interior-light scene).**  v7 sweep result on the 66-cell displaced-painter grid (PT, 4 spp): snell+T2 mean ok rate **94.4 %**, mean ratio Œ£Lsms/Œ£Lsup **0.893**; uniform+T2 mean ok **43.4 %**, ratio **0.351**.  Pre-T2 baseline (post seed-overtrace fix only) was ok ~67 %, ratio ~0.85 for snell.  Snell+T2 is +27 pp ok / +0.04 ratio over pre-T2 snell, cleanly beats uniform on heavy-displaced refractive caustics; uniform+T2 is competitive (or wins) on smooth analytical caustics (`sms_k2_glasssphere`: uniform 93 % ok / ratio 5.81 vs snell 86 % ok / ratio 6.60).  Recommended in BOTH snell and uniform mode for any scene whose natural caustic K is known ‚Äî the `sms_seeding` regime guidance still holds: snell for displaced refractive, uniform for smooth analytical.

- **SMS uniform-mode chain-length cap (2026-05)** ([docs/SMS_SEEDING_OVERTRACE_FIX.md](docs/SMS_SEEDING_OVERTRACE_FIX.md) ¬ß"Uniform-mode follow-up"): the seed-overtrace fix below introduced an emitter-projection cap inside `BuildSeedChain` / `BuildSeedChainBranching` / `SnellContinueChain` that's correct in `sms_seeding "snell"` (passes the LIGHT as `end`) but WRONG in `sms_seeding "uniform"` (passes the uniform-area sampled point on the caster as `end`).  Cap fired at `|sp ‚àí pos| √ó 1.05`, clipping the seed chain to a single specular vertex when the natural caustic is k=2.  Combined with RISE not enforcing Mitsuba's `m_config.bounces` target chain length (we have `sms_max_chain_depth` as a *maximum*, default 10), uniform mode produced wildly variable chain lengths (k=1, 2, 3, 4) and a 36‚Äì53 % phys-fail rate vs snell's 3‚Äì17 %.  **Two-part fix**: (a) added `applyEmitterStop` parameter to the seed builders, default `true` for back-compat, with all six uniform-mode call sites passing `false`; (b) **scene-authoring rule** ‚Äî for uniform mode, set `sms_max_chain_depth` to the natural caustic K (typically 2 for a glass shell or interior-light scene), not the default 10.  Superseded by the `sms_target_bounces` entry above which is the cleaner config ‚Äî keep `sms_target_bounces` set rather than hijacking `sms_max_chain_depth`.

- **SMS seed-overtrace fix (2026-05)** ([docs/SMS_SEEDING_OVERTRACE_FIX.md](docs/SMS_SEEDING_OVERTRACE_FIX.md)): `BuildSeedChain` and `BuildSeedChainBranching` were collecting specular hits **past the emitter** when the emitter was inside a closed dielectric shell (e.g. interior light in the Veach-egg cavity).  The natural caustic chain is `k=2` (wall ‚Üí outer-egg ‚Üí inner-cavity ‚Üí light), but the trace continued past the light through inner-front and outer-front, producing a `k=4` chain whose v2/v3 were on the opposite side of the emitter.  Newton converged to algebraically valid roots where wi/wo were both on the same side of the surface ‚Äî geometric reflection, not refraction ‚Äî and `ValidateChainPhysics` correctly rejected them.  This was the dominant `physicsFail` source on heavy-displaced scenes and the misdiagnosis behind multiple prior "structural multi-modality" conclusions.  **Fix**: stop adding seed vertices once a candidate hit's projection along the original `(start ‚Üí end)` direction exceeds `totalDist √ó 1.05` (the 5% margin handles refractive bending).  ~20 lines, two sites in `ManifoldSolver.cpp`.  **Result on the 66-cell displaced-painter sweep**: Œîok% mean +21.6 pp (range +10 to +35), Œîphys% mean -17.3 pp (phys-fails now ~0% at low disp, ~3% at disp=40), ratio Œ£Lsms/Œ£Lsup mean +0.225.  No regression on the canonical SMS scenes (`sms_k1_refract`, `sms_k2_glassblock`, `sms_k2_glasssphere`, `sms_k1_botonly`).  **What it does NOT change**: residual ~30% Newton-fail rate (still plateau-stalled at ‚ÄñC‚Äñ ‚àà [0.1, 1) ‚Äî the documented basin-of-attraction limit of plain Newton).  Rewrites several prior conclusions: the "spurious basin is topologically separated" claim from the F+E+G investigation was wrong (the seed was driving Newton there); photon-seeding's null result (G) was contaminated by the same seed-builder bug and warrants re-test on the fixed baseline.  Diagnostic instrumentation (sign-product, seed-validity, post-Newton ‚ÄñC‚Äñ, chain-length, cascade-depth, surface-angle, perturbed-restart counters) is gated under `SMS_SOLVE_DIAG` (default 0); flip to 1 for future SMS regressions.

- **SMS Levenberg-Marquardt + scene-authoring rule for displaced caustics** ([docs/SMS_LEVENBERG_MARQUARDT.md](docs/SMS_LEVENBERG_MARQUARDT.md)): NOTE ‚Äî the "root cause = constraint multi-modality" diagnosis below is **superseded** by the seed-overtrace fix above; the dramatic SMS power drop turned out to be a seeding bug, not a deep solver / constraint-landscape problem.  LM's measured benefit (~5 pp on Newton-fails) is still real but on a much smaller residual gap than reported here.  `ManifoldSolver::NewtonSolve` damps the Jacobian's diagonal by `Œª √ó mean(|J_ii|)` between Newton iterations, with Œª shrinking on accepted line-search steps (toward pure Newton) and growing on rejected ones (toward gradient descent).  Runtime toggle: `SMSConfig::useLevenbergMarquardt` / `ManifoldSolverConfig::useLevenbergMarquardt` (bool, **default FALSE ‚Äî opt-in**); wired through `RISE_API_Create*Rasterizer` factories.  Off by default because LM is ~50-100% slower than pure Newton on heavy-displacement scenes for ~5pp Newton-fail improvement ‚Äî the cost-vs-gain doesn't justify a global default.  Set true at API construction time for heavy-displaced-mesh scenes that need the extra robustness.  No scene-language parser hook yet.  Variant: damped Newton on the original `J`, not full Marquardt-style normal equations ‚Äî keeps the existing block-tridiagonal solver path.  **Root cause of the dramatic SMS power drop on heavily-displaced meshes is `ValidateChainPhysics` rejecting algebraically-valid-but-physically-nonsensical converged chains** ‚Äî not Newton convergence failures (Pearson r = ‚àí0.914 between phys-fail rate and PT/VCM energy ratio across a 66-point grid; Newton-fail correlation is only ‚àí0.258, barely signal).  Smooth-displaced surfaces create a half-vector constraint landscape with many local minima, some of which correspond to chains where wi/wo are on the wrong sides of the surface for the labeled reflection/refraction topology.  Newton minimises ||C|| blindly and lands on those spurious minima.  **Heaviside displacement (step function, no smooth slopes) preserves energy at ANY amplitude** because its surface is mostly flat with one sharp boundary ‚Äî only one real local minimum per seed.  Switching to a single-mode low-frequency painter (Gerstner with `num_waves=2 median_wavelength=1.0`) on the same scene at disp_scale=10: ok rate **23 % ‚Üí 49 %**, energy ratio **0.41 ‚Üí 0.71**, and the curve becomes invariant to tessellation detail.  Note: a controlled-smoothness experiment (`controlled_smoothness2d_painter`, 6 boundary continuity orders from C‚Åª¬π Heaviside to C^‚àû Gaussian, matched amplitude/radius) refuted the naive "smoother painter = fewer Newton failures" hypothesis ‚Äî all six modes cluster in 45-53 % ok% with no monotonic relationship to smoothness order.  Heaviside (worst continuity) actually has the HIGHEST ok rate because flat-region Newton converges trivially.  Practical rule: prefer FEWER bumps with longer wavelengths over many small bumps regardless of continuity; finer tessellation does NOT help and often hurts.  Per-Solve Œª is local-stack (no global state, no thread races) and starts at 0 each call so the first iter is pure Newton ‚Äî preserves baseline behaviour on well-conditioned chains.  Measured impact on the displaced Veach egg sweep (snell, M=1, no two-stage, no photons, disp 0..10): Newton-fail rate drops 4‚Äì10 pp at every disp value (e.g. disp=0 23.1% ‚Üí 18.1%, disp=10 72.0% ‚Üí 67.4%); 15‚Äì30k chains rescued per disp.  Smooth-analytic regression (sms_k2_glasssphere) gains +5.8 pp ok rate; plateau-bound scenes (sms_k1_refract, sms_k2_glassblock) unchanged within MC noise.  Cost: ~50-100% slower on heavy-displacement scenes (LM escalations consume the iter budget); negligible on smooth scenes / scenes without SMS.  Two regimes LM **cannot** fix: multi-basin landscapes with no descent connection from the seed (need better seeding ‚Äî photons in the right basin), and C¬π discontinuities at triangle edges (need smoother surface evaluation).

- **SMS photons + heavy-displacement Newton plateau** ([docs/SMS_PHOTON_DECOUPLING_AND_DISPLACEMENT_LIMIT.md](docs/SMS_PHOTON_DECOUPLING_AND_DISPLACEMENT_LIMIT.md)): two correctness fixes plus the documented Newton-plateau limit on displaced caustics.  (1) Photon emit pre-populates the IOR stack via `IORStackSeeding::SeedFromPoint` ‚Äî without this, photons emitted from a luminaire inside nested dielectrics die on bounce 1 (verified 0% ‚Üí 81% photon-storage rate on the displaced Veach egg).  (2) Snell-mode SMS no longer gates `QuerySeeds` on `multi_trials > 1`; photon retrieval is independent of `multi_trials`, and `totalTrials` extends to absorb every queried photon ‚Äî previously a scene that set only `sms_photon_count > 0` paid the photon-pass cost and never used a single seed.  (3) `maxPhotonSeedsPerShadingPoint` (default 16, Fisher-Yates random subsample at all 4 photon-consumption sites) bounds per-pixel Newton-solve cost so the decoupled photon path doesn't pay 100√ó overhead in dense kd-tree regions.  Fundamental finding: the disp 0 ‚Üí disp 10 power drop on the Veach egg is **not** an iteration-budget issue (`maxIter=200` is identical to `maxIter=30`) and **not** a seed-quality issue (`seedTooFar=0%` everywhere) ‚Äî Newton stalls on plateaus where ||C|| ‚àà [0.1, 1.0) (74-85% of Newton-fails).  This is the documented basin-of-attraction limit of plain Newton on a displaced normal field.  Doc captures three avenues that would actually move the needle: robust solver (LM / trust-region), photon seeds + robust solver, or specular polynomials all-roots enumeration.  None are in master yet.

- **BDPT no longer participates in SMS (excised 2026-05-07)**: SMS was wired into BDPT via `EvaluateSMSStrategies{,NM}` (per-eye-vertex specular-chain evaluation) plus a cross-strategy emission-suppression predicate `ShouldSuppressSMSOverlap` that prevented BDPT's (s==0) emission strategy from double-counting SMS-reachable caustic paths.  Both were removed.  Reasons: state-of-the-art renderers (Mitsuba 3, PBRT) wire SMS only into unidirectional PT ‚Äî the BDPT-with-SMS combination is not in the reference implementations; the `ShouldSuppressSMSOverlap` predicate added structural MIS-walk complexity for no measurable variance reduction over PT+SMS on the same scenes.  PT, MLT still use SMS (`pathtracing_*_rasterizer` + `mlt_spectral_rasterizer` factories continue to take `ManifoldSolverConfig`); VCM never used SMS (handles caustics via merging).  **ABI / scene-file break**: `RISE_API_CreateBDPT{Pel,Spectral}Rasterizer` no longer take the 11 `sms_*` parameters; the BDPT chunk parsers no longer accept `sms_*` lines (legacy scenes hard-fail with "ChunkParser:: Failed to parse parameter name `sms_enabled` (not declared in `bdpt_pel_rasterizer` descriptor)").  Scene migration: switch the integrator chunk from `bdpt_pel_rasterizer` ‚Üí `pathtracing_pel_rasterizer` (or strip the `sms_*` lines if SMS wasn't actually needed for that scene).  Existing BDPT+SMS test scenes were converted: `sms_slab_close_bdpt_sms.RISEscene` ‚Üí `sms_slab_close_pt_sms_hispp.RISEscene` (PT+SMS at samples=64), the spectral twin `spectral_dispersive_caustic_bdpt_sms.RISEscene` was deleted (the PT version `spectral_dispersive_caustic_pt_sms.RISEscene` already provided coverage), and `tests/BDPTSMSSuppressionTest.cpp` was removed.

- **Path-tree branching at multi-lobe delta vertices (excised 2026-05)**: PT, BDPT, VCM, MLT, and uniform-mode SMS used to deterministically split the path at Fresnel splits (`branching_threshold` knob, default 0.5 ‚Üí 1.0 ‚Üí removed entirely).  An investigation on `sms_test_mixed_chain` (CLAUDE companion `BDPTPelRasterizer.cpp` per-pixel instrumentation, 2026-05) found that BDPT's per-vertex `pdfFwd` recorded during a deterministic-split walk drives `MISWeight`'s ratio chain incorrectly for s‚â•1 strategies that interact with branched eye chains ‚Äî local mean bias of ~2.3% on SDS-chain regions vs PT and BDPT-no-branch (which agree within MC noise).  PT branching was mathematically unbiased but gave only ~3% std reduction at ~30-50% extra compute per sample (per-second variance is worse than stochastic single-scatter selection).  All integrators now use stochastic single-lobe selection at multi-lobe vertices with proper `kray / selectProb` correction.  SMS uniform mode falls back to single-chain stochastic seed with `multi_trials` for variance reduction on multi-modal scenes ‚Äî matches Mitsuba SOTA convention.  Snell-mode SMS (default) was unaffected.  Scenes that previously set `branching_threshold` will fail to parse ‚Äî the parameter has been removed.  The retired `branch` param on PathTracing/DistributionTracing shader-ops is silently ignored by the parser; photon tracers keep their own `bBranch` (out of scope, separate mechanism).
- **JH spectral-uplift LUT (`extlib/jakob-hanika-luts/rec709.coeff`) ships at ~3.9 % gamut-edge cell failures ‚Äî the residual is the JH sigmoid model's expressive limit, not a colour-space problem** ([docs/JH_LUT_GAMUT.md](docs/JH_LUT_GAMUT.md)).  Stage A of the 2026-05-24 colour-space migration retrained the LUT against Rec.709 Linear (D65); failures dropped from 22 % (ROMM) to 3.9 % (mean residual 1.2 √ó 10‚Åª¬≥, max 9.5 √ó 10‚Åª¬≤), all concentrated in the deep-blue gamut corner where the sigmoid form has limited expressive power.  The LUT generator (`tools/JakobHanikaLUTGen.cpp`) is parameterized via `--target {rec709|romm|acescg}` so a future ACES migration is a one-line bake change; the runtime accessor `RGBToSpectrumTable::Get()` converts callers into `LUTTargetPel` (typedef'd to `Rec709RGBPel` post Stage A) at the call boundary, with `RGBUnboundedSpectrum::FromRGB` / `RGBIlluminantSpectrum::FromRGB` doing scale + normalize in LUT-target space.  The bake script `tools/GenerateSpectrumLUTHeader.py` hard-fails on `--target` ‚âÝ `rec709` until the runtime conversion type is extended.  Reference SPD for `RGBIlluminantSpectrum` is **D65** (matching Rec.709 whitepoint) ‚Äî pre-migration was D50.

- **`RISEPel = Rec709RGBPel` since Stage B of the 2026-05-24 colour-space migration** ([docs/COLOR_SPACE_MIGRATION.md](docs/COLOR_SPACE_MIGRATION.md)).  Pre-migration was `ROMMRGBPel` (D50, ProPhoto primaries ‚Äî wide gamut but primaries outside the spectral locus ‚Üí 22 % JH LUT corner failures; D50 whitepoint forced Bradford CATs on every CMF-vs-LUT integration).  Post-migration: sRGB / BT.709 primaries, D65 whitepoint, industry-standard PBR working space; `FilteredFilm` / `ProgressiveFilm` resolve XYZ ‚Üí RISEPel via `XYZtoRec709RGB` (D65 ‚Üí D65, matrix-only, no Bradford).  `AP1RGBPel` (ACEScg) is pre-staged with full conversion-constructor surface for a future ACES migration; the architectural cost reduces to: flip the typedef in `Color.h`, flip `LUTTargetPel` in `RGBToSpectrumTable.h`, retrain LUT with `--target acescg`, audit the same Stage B sites listed in COLOR_SPACE_MIGRATION.md.  **Scene-file authoring note**: `color_space ROMMRGB_Linear` on a `png_painter` / image_painter chunk no longer means "verbatim store" ‚Äî it now applies a real Rec.709 ‚Üí ROMM Bradford+matrix conversion (warps normal-map vectors).  The new verbatim-store idiom is `color_space Rec709RGB_Linear`.  Use `tools/migrate_scenes_color_space_b.py` to convert image-painter contexts (preserves `colorspace ROMMRGB_Linear` on `uniformcolor_painter` and `color_space ROMMRGB_Linear` on `file_rasterizeroutput`, where the user's ROMM intent is still honoured via real conversion at the boundary).  `IntegratorXYZtoROMMRGB` no longer exists ‚Äî all callers were eliminated.  `FrameStoreColorSpace::FSColorSpace::sRGB_Linear` is now the identity-to-RISEPel target; `ROMM_Linear` does a real Rec.709 ‚Üí ROMM conversion.

- **Chunk parsers are descriptor-driven** (since 2026-04). Every `IAsciiChunkParser` overrides only `Describe()` and `Finalize(const ParseStateBag&, IJob&)`; the default `ParseChunk` impl validates input lines against `Describe().parameters` and dispatches to `Finalize`. The descriptor IS the accepted-parameter set, so syntax-highlighter / suggestion-engine drift is structurally impossible. Adding a new chunk = new `IAsciiChunkParser` subclass + register in `CreateAllChunkParsers()`. Adding a parameter = one entry in `Describe()` + one `bag.GetX(...)` call in `Finalize`. Full how-to in [src/Library/Parsers/README.md](src/Library/Parsers/README.md); architecture overview in the header of [src/Library/Parsers/AsciiSceneParser.cpp](src/Library/Parsers/AsciiSceneParser.cpp).

## Thread priority ‚Äî read before touching anything threading-related

- **Production default (topology-aware):** every P-core gets a render worker, every E-core except **one** also gets a worker. The reserved E-core keeps UI / daemons responsive. macOS workers use `QOS_CLASS_USER_INITIATED`; Linux/Windows use CPU-affinity pinning.
- **Benchmarks:** set `render_thread_reserve_count 0` to give every core a worker. `./bench.sh` at repo root does this automatically. Without the override, Apple Silicon's E-core reservation makes the machine appear ~10 % slower than it really is.
- **Do NOT** put all threads at `QOS_CLASS_UTILITY` / `BELOW_NORMAL` / `nice(10)`. That's the old bug ‚Äî cost 2‚Äì4√ó real throughput on macOS. A legacy `force_all_threads_low_priority` option still exists for users who explicitly want the background-render mode; it's opt-in only.
- Controls live in `src/Library/Utilities/CPUTopology.{h,cpp}`, `src/Library/Utilities/ThreadPool.cpp`, `src/Library/Utilities/Threads/ThreadsPTHREADs.cpp`, `src/Library/Utilities/Threads/ThreadsWin32.cpp`.
- See [docs/PERFORMANCE.md](docs/PERFORMANCE.md) "Thread priority policy" for the full rationale, per-platform APIs, and measurement recipe.
