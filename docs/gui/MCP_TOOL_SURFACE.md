## 0. One-paragraph thesis

RISE is unusually well-suited to be an MCP server because the two hard problems of agentic tooling are already solved in-tree: the scene is **validatable text**, and the grammar is **machine-introspectable** (`SceneGrammar::Instance()` self-describes every chunk and parameter ŌĆö the same source of truth that drives the parser, the syntax highlighters, and the autocomplete engine, with drift "structurally impossible" per [../../src/Library/Parsers/README.md](../../src/Library/Parsers/README.md)). The MCP server is therefore a **thin JSON-RPC adapter** over three things RISE already owns: (1) the grammar descriptor (ŌåÆ tool/resource schemas), (2) the `SceneEditController` mutation/render brain (ŌåÆ tools), and (3) the introspection + `FrameStore` surfaces (ŌåÆ resources). The bulk of the new C++ is the protocol plumbing and **two genuinely new capabilities** ŌĆö a side-effect-free `validate(scene_text)` parse pass and an in-memory scene-text accessor ŌĆö both small and both useful beyond MCP.

## 1. Background ŌĆö what MCP gives us, and which RISE pieces map to it

The [Model Context Protocol](https://modelcontextprotocol.io/) is a JSON-RPC 2.0 standard (current spec revision `2025-11-25`) with three first-class primitives. We use all three:

| MCP primitive | Control model (per spec) | RISE usage |
|---|---|---|
| **Resources** ŌĆö read-only context, addressed by URI ([spec/server/resources](https://modelcontextprotocol.io/specification/2025-11-25/server/resources)) | application-driven (host decides what to include) | scene text, grammar schema, scene-graph, **rendered framebuffer + AOVs**, render stats / log / variance / auto-router decision, asset listings (┬¦3) |
| **Tools** ŌĆö model-invoked actions with JSON-Schema `inputSchema` ([spec/server/tools](https://modelcontextprotocol.io/specification/2025-11-25/server/tools)) | model-controlled (human-in-the-loop SHOULD gate) | mutate / add / clone / material / HDRI / camera / render-control / save / export / **`validate`** (┬¦4) |
| **Prompts** ŌĆö reusable templates, user-selected ([spec/server/prompts](https://modelcontextprotocol.io/specification/2025-11-25/server/prompts)) | user-controlled (slash commands) | a handful of curated workflows: "relight for golden hour", "diagnose dark render", "make N lighting variants" (┬¦4.7) |

Why MCP rather than a bespoke protocol (settles roadmap ┬¦9.2 rationale): it is the emerging standard for LLM tool use, it is **provider-agnostic** (decouples the server from Claude vs Gemini vs a local model ŌĆö the provider dialect translation lives in the agent runtime, not here), and it gives **external-client reuse for free** ŌĆö Claude Desktop, IDEs, and Claude Code can drive RISE with zero extra server work.

### 1.1 The three RISE surfaces the server wraps

1. **`SceneGrammar`** ([../../src/Library/SceneEditorSuggestions/SceneGrammar.h](../../src/Library/SceneEditorSuggestions/SceneGrammar.h)) ŌĆö `SceneGrammar::Instance()` aggregates every `ChunkDescriptor` from the parser registry. A `ChunkDescriptor` ([../../src/Library/Parsers/ChunkDescriptor.h](../../src/Library/Parsers/ChunkDescriptor.h)) carries `keyword`, `category`, `description`, and a `std::vector<ParameterDescriptor>`. Each `ParameterDescriptor` carries `name`, `kind` (`ValueKind` enum), `required`, `repeatable`, `enumValues`, `referenceCategories`, `tupleKinds`, `presets`, `description`, `defaultValueHint`, and `unitLabel`. **This is the schema source.** (┬¦5)
2. **`SceneEditController`** ([../../src/Library/SceneEditor/SceneEditController.h](../../src/Library/SceneEditor/SceneEditController.h)) ŌĆö the cross-platform editor brain: the one mutation path. Already exposes `SetProperty` / `SetPropertyForCategory`, `SetSelection`, `CloneActiveCamera`, `Undo` / `Redo`, `RequestProductionRender`, `RequestSave`, `Start` / `Stop`, the accordion/introspection accessors, and the live-preview render thread. Constructed via `RISE_API_CreateSceneEditController` ([../../src/Library/RISE_API.h](../../src/Library/RISE_API.h)).
3. **Introspection + framebuffer** ŌĆö `CameraIntrospection` / `ObjectIntrospection` / `LightIntrospection` / `MaterialIntrospection` / `MediaIntrospection` / `FilmIntrospection` / `RasterizerIntrospection` ([../../src/Library/SceneEditor/](../../src/Library/SceneEditor/)) all return `std::vector<CameraProperty>` rows (`name` / `kind` / `value` / `description` / `editable` / `presets` / `unitLabel`); `FrameStore` ([../../src/Library/Rendering/FrameStore.h](../../src/Library/Rendering/FrameStore.h)) holds the rendered beauty pass plus typed AOV channels (`Channel.h`: `Beauty`, `Alpha`, `Albedo`, `Normal`, `Depth`, `ObjectId`, `PrimitiveId`); `FileRasterizerOutput` ([../../src/Library/Rendering/FileRasterizerOutput.h](../../src/Library/Rendering/FileRasterizerOutput.h)) writes PNG/EXR/HDR/TIFF.

### 2.1 The server is library code (principle #2)

A new subsystem **`src/Library/Agent/`** holds the entire server. **Type/dir names avoid the bare `MCP` token** (per [../GUI_ROADMAP.md](../GUI_ROADMAP.md) ┬¦16 ŌĆö `MCPClientConnection` already means Master Control Program); the protocol is still MCP, but the C++ surface is `Agent*`/`Scene*`-prefixed:

**Single `SceneEditController` instance per open document.** Both the in-app client and external clients drive the *same* controller, so an AI edit from Claude Desktop and a gizmo drag in the GUI both flow through `SceneEditController` ŌĆö specifically a `Commit(SceneTransaction)` (TRANSACTION_MODEL ┬¦4) that wraps the shipped `Apply`. Both land in the same undo stack (attributed by originating session, TRANSACTION_MODEL ┬¦8) and show up in the same "show me the code" panel. This is principle #6 made literal: there is no parallel write path for AI.

**Concurrency.** `SceneEditController` already runs a render thread and serializes edits against it with `mMutex` + a cancel-restart machine. The agent server adds one more producer of edits. Tools that mutate **must marshal onto the UI/owner thread** (the thread that constructed the controller), exactly as the platform bridges already do ŌĆö the controller's contract is "edits happen on the owner thread." Concretely: the in-process transport posts a closure to the host's run loop; the loopback-HTTP and stdio transports run their socket I/O on a server-owned thread but hand each mutating tool call to the controller via a thread-safe command queue drained on the owner thread. Read-only tools/resources read a **published immutable `SceneSnapshot`** (never the live graph) and can answer on the server thread (TRANSACTION_MODEL ┬¦6.2). **v1 is single-writer (TRANSACTION_MODEL ┬¦0 #4):** the in-app GUI and the in-process agent-policy commit; an **external MCP client is read + propose-only** ŌĆö its mutating `tools/call` does not commit directly but stages a `Propose` (one-shot whole-edit-set, TRANSACTION_MODEL ┬¦12) that the document owner applies. **This split is the load-bearing distinction ┬¦4.0 makes precise: a mutating tool resolves to one of two contracts ŌĆö a *committed* tool (owner/in-app session) or a *proposed* tool (external/read-only session) ŌĆö and only a *committed* tool returns `newEpoch`/`undoAvailable`; a *proposed* tool returns a proposal ID + status (┬¦4.0, ┬¦8). The ┬¦4.1ŌĆō┬¦4.5 tool tables describe the *committed* contract's handler (it calls `SceneEditController::Commit`); the *proposed* contract wraps the same edit set in a `Propose` instead.** **Edits are reconciled optimistically, not by locking:** each mutating call carries the `baseEpoch` it read at, and a commit (or the owner-approval of a proposal) whose precondition lost the race is **rejected with a conflict** (never a silent clobber) ŌĆö the full `DocumentId`/conflict contract is [TRANSACTION_MODEL.md](TRANSACTION_MODEL.md) ┬¦4.4. **Render arbitration is likewise delegated:** a render tool does not stop/restart the viewport itself; it submits a job to the single `RenderCoordinator` ([RENDER_COORDINATOR.md](RENDER_COORDINATOR.md) ┬¦2.2), which holds the one render slot and decides preempt-vs-queue-vs-reject.

### 2.3 Transport choice (resolves GUI_ROADMAP ┬¦13 open question #3)

The spec defines two standard transports ŌĆö **stdio** (client launches server as a subprocess, newline-delimited JSON-RPC over stdin/stdout, stderr for logs) and **Streamable HTTP** (single endpoint, POST for messages, optional GET+SSE for serverŌåÆclient streaming, sessions via `MCP-Session-Id`) ([spec/basic/transports](https://modelcontextprotocol.io/specification/2025-11-25/basic/transports)). RISE supports **three** transports behind the `AgentTransport` interface, because the embedding constraint differs by client:

| Transport | Used by | Why | Notes |
|---|---|---|---|
| **In-process** (`AgentInProcessTransport`) | the in-app chat panel | No socket, no serialization round-trip across a process boundary; the in-app MCP *client* and the server share the address space and the live `SceneEditController`. Lowest latency for the seeŌåÆeditŌåÆrenderŌåÆsee loop. | Still goes through the full MCP message layer (so the in-app client is a real MCP client, not a bespoke shortcut) ŌĆö just over a memory queue instead of a socket. Keeps the in-app path honest and identical in behavior to external clients. Authenticated by construction (shared address space); the `Origin`/token checks below apply only to loopback-HTTP (AI_SECURITY_MODEL ┬¦13 open-q 4). |
| **Loopback HTTP** (`AgentLoopbackHttpTransport`) | external clients that attach to a *running* GUI instance (the headline: "Claude Desktop drives my open RISE window") | Streamable HTTP is the spec's recommended networked transport and the only way an already-running engine instance can accept a second client. Bound to **127.0.0.1 only**. | **Security (mandatory, per spec ŌĆö full design in [AI_SECURITY_MODEL.md](AI_SECURITY_MODEL.md) ┬¦8):** validate the `Origin` header (HTTP 403 on mismatch) to prevent DNS-rebinding; bind to loopback, never `0.0.0.0`; require a per-launch bearer token printed in the GUI ("Enable AI access" surfaces the URL + token); any session ID is secure and **not** used for auth. Port is ephemeral, advertised to the user for manual client config. |
| **stdio** (`AgentStdioTransport`) | external clients that want RISE to be a *headless* MCP server they spawn (Claude Code, CI, scripted batch) | The spec's most-interoperable transport; clients "just work" by launching `rise --agent-stdio scene.RISEscene`. No port, no token ŌĆö the OS process boundary is the trust boundary (AI_SECURITY_MODEL ┬¦8.1.6). | Spawns a headless engine (no GPU present surface) with a `SceneEditController` whose preview sink is null but whose `RequestProductionRender` + `FrameStore` work. `stderr` carries `RISE_Log.txt`-style logs; `stdout` is MCP-only (spec requirement). **Headless has no GUI owner to approve a `Propose`, so it runs under the explicit headless autonomy policy (┬¦4.0): `read`+`validate` only by default, and the *committed* contract only under an operator-set `--agent-autonomy=commit` launch grant.** |

**Coexistence model (the crux of open question #3).** The in-app client (in-process) and external clients (loopback-HTTP) can be attached to the **same** engine instance simultaneously. They are distinct MCP *sessions* but share one `SceneEditController` and one undo stack (entries **attributed to the originating client**, TRANSACTION_MODEL ┬¦8). The server serializes commits through the owner-thread command queue (┬¦2.2) and the controller's `mMutex`, so they are *totally ordered* (TRANSACTION_MODEL ┬¦6.2, M1). **It is NOT "last edit wins":** v1 is single-writer (TRANSACTION_MODEL ┬¦0 #4) ŌĆö the in-app session commits its own gestures, and an external session is **propose-only** (its edits stage as a `Propose` the owner applies). A commit (or owner-approval) whose `baseEpoch` lost the race is **rejected with a conflict**, never silently overwritten; the loser re-reads `rise://scene/*` and rebases. Each session gets its own `resources/subscribe` set and its own `notifications/resources/updated` stream (e.g. external client subscribed to `rise://framebuffer` gets pinged when the in-app user kicks a render). The headless stdio case is the *opposite* topology ŌĆö one server, typically one client, no GUI ŌĆö and never coexists with an in-app client by construction; **with no GUI owner to approve a `Propose`, it does not use the propose-only contract at all but runs under the headless autonomy policy (┬¦4.0): read+validate by default, committed-contract only under an operator launch grant.**

### 2.4 Android note (principle #3)

Android consumes the identical `AgentServer` over JNI. The Java/Kotlin shell embeds the **in-process** transport exactly like the desktop in-app client ŌĆö there is no Android-specific server. The chat surface is mobile-native (a Compose chat view is the natural, arguably *best*, surface for "talk to your renderer from your phone" per roadmap ┬¦10.4 Tier A). External attach on Android is **adb-forwarded loopback only**: a laptop client reaches the phone via `adb forward` of the loopback port over USB, which terminates on the phone's `127.0.0.1` and thus satisfies the loopback-bind + `Origin` rules. **There is no same-LAN transport** ŌĆö same-LAN would require binding a non-loopback interface, which the loopback-only rule forbids on every platform ([AI_SECURITY_MODEL.md](AI_SECURITY_MODEL.md) ┬¦8.2 explicitly corrects the earlier "or same-LAN" wording). adb-loopback-HTTP is a developer/Tier-C nicety (token-gated like desktop), not a launch requirement. The only JNI additions are: start/stop the server, hand it the JNI-owned `SceneEditController`, and marshal mutating tool calls onto the Android main thread (the same thread-affinity contract). Because the server is shared C++, Android gets the entire tool/resource catalog for free ŌĆö the roadmap's central payoff.

## 3. Resource catalog

Resources are read-only and addressed by URI under a custom `rise://` scheme (the spec explicitly permits custom schemes that follow RFC 3986). The server advertises `resources: { subscribe: true, listChanged: true }`. Static resources appear in `resources/list`; parameterized ones appear in `resources/templates/list` as RFC 6570 URI templates.

| URI | Kind | MIME | Shape | Backed by | Subscribe? |
|---|---|---|---|---|---|
| `rise://scene/text` | static | `application/json` | **Structured payload `{ text, documentId: {uuidHi,uuidLo}, revision, redactions[] }`** ŌĆö NOT bare `text/plain` (the contract correction, ┬¦3.1). `text` is the full current scene as `.RISEscene` (comments, `FOR`, `DEFINE` intact), **redacted for egress** (AI_SECURITY_MODEL ┬¦9.1.5) but **length-preserving** ŌĆö each secret byte is replaced by a **single-byte ASCII mask byte** (e.g. `*`, U+002A ŌĆö one byte, **not** the multi-byte `ŌĆó`/U+2022 which is 3 UTF-8 bytes and would shift every following offset; ┬¦3.1) so `text`'s byte offsets are identical to the canonical document the engine validates and diffs against; `redactions[]` carries each `{ offset, length }` out-of-band so the model knows where bytes were masked without seeing them. `documentId` + `revision` are the observed `DocumentId = (UUID, revision)` (TRANSACTION_MODEL ┬¦3.4, E2) a client bases its `baseDocument` precondition on; **the `text`/`EditScript` and the scene graph are two faces of one document revision, so this `documentId` describes graph+text atomically (┬¦3.1).** | new in-memory scene-text accessor (┬¦3.1) | yes ŌĆö fires on any committed edit |

### 3.1 In-memory scene-text accessor + the structured, byte-offset-stable resource contract (genuinely new, small)

**Finding (resource-contract impossibility ŌĆö second-review ┬¦3; the mask-byte width ŌĆö GUI_ROADMAP ┬¦13a #4):** the original `rise://scene/text` design declared `text/plain` *and* required the payload to carry an epoch *and* be run through `SecretRedactor`. Those three are mutually inconsistent. A `text/plain` body has nowhere to carry the `DocumentId` a `baseEpoch` precondition needs; and na├»ve secret redaction (replacing `sk-abc123` with `[REDACTED]`) **changes the byte count and shifts every subsequent offset**, so the text the model reads would no longer be the text the engine validates and diffs against ŌĆö a `validate`/diff computed from the model's copy would mis-locate every span after the first redaction. **And even a "one mask glyph per secret byte" scheme breaks offset-faithfulness if the glyph is multi-byte: the bullet `ŌĆó` is U+2022 = three UTF-8 bytes, so masking an N-byte secret with N bullets writes 3N bytes and shifts every following offset by 2N.** The mask must therefore be a **single-byte** code unit. **Resolution: the resource is structured data with a length-preserving redaction contract that masks with a single-byte ASCII mask byte.**

**The contract (the byte-offset rule, made explicit):**

1. **Structured envelope, not bare text.** `rise://scene/text` returns `application/json`: `{ text, documentId: { uuidHi, uuidLo }, revision, redactions: [ { offset, length }, ŌĆ” ] }`. The `documentId` + `revision` are the observed `DocumentId = (UUID, revision)` (TRANSACTION_MODEL ┬¦3.4); a client copies the **pair** into a mutating tool's `baseEpoch` (┬¦4 / ┬¦4.1). `rise://scene/chunk/{name}` uses the same envelope plus `spanOffset`.
2. **Length-preserving redaction with a single-byte mask.** Each secret in `text` is overwritten **byte-for-byte** with a fixed **single-byte ASCII mask byte** ŌĆö `*` (U+002A, one UTF-8 byte). It must **not** be the bullet `ŌĆó` (U+2022 = 3 UTF-8 bytes), which would write 3 bytes per masked byte and shift every following offset (the ┬¦-finding above). With a single-byte mask, `len(redacted) == len(canonical)` and **every byte offset in `text` equals the corresponding offset in the canonical document the engine holds.** A diff or `validate` the model computes against `text` therefore lines up exactly with what a mutation tool will splice (this is the same span discipline the round-trip `SourceSpanIndex` and the "show me the code" diff rely on, TRANSACTION_MODEL ┬¦10). The `redactions[]` array reports each masked `{ offset, length }` out-of-band so the model can *see that* bytes were hidden (and avoid "fixing" a `****` run) without ever seeing the secret itself.
3. **Redaction is egress-only.** This masking happens on the resource-read path (an egress boundary, AI_SECURITY_MODEL ┬¦9.1.5) ŌĆö it is **never** applied to the canonical in-memory document or to `SaveEngine`'s on-disk write (AI_SECURITY_MODEL ┬¦9.1.3, which scopes redaction to egress and explicitly excludes `SaveEngine`). The model gets a masked-but-offset-faithful view; the file keeps its real bytes.
4. **Redacted text must never be written back wholesale (the round-trip hazard, GUI_ROADMAP ┬¦13a #4).** Because the model only ever sees the masked `text` (real secret bytes replaced by the mask byte), feeding that masked text back as a full-document replacement would **overwrite the real secrets in the canonical document with mask bytes** ŌĆö silently destroying a key the user deliberately keeps in their scene. Therefore `apply_scene_text` (┬¦4.5) **forbids wholesale replacement whenever the prior read carried any `redactions[]`** (refused with a structured `REDACTED_WHOLESALE_REPLACE` error, not silently applied). Three sanctioned ways to edit a document that contains secrets, in preference order: **(a)** prefer **scoped/structured tools** (`set_property`, `set_transform`, the ┬¦4.1ŌĆō┬¦4.3 ops) and span-targeted edits that never touch a masked region ŌĆö server-side span edits splice only the model-named span and preserve the opaque `redactions[]` bytes verbatim; **(b)** if a wholesale path is unavoidable, accept the placeholder text only with a **crypto-bound redaction map** (each mask run keyed to the original bytes by a per-read nonce the server holds; on apply the server restores the original bytes for any unchanged mask run server-side, and rejects the call if a mask run was altered or moved) so the masked bytes are reconstituted, never persisted; **(c)** otherwise refuse and tell the model to re-read and use scoped edits. The default (and the only v1-required path) is **(a)** ŌĆö (b) is a documented option, not a v1 obligation.
5. **The source text and the scene graph are one document revision (atomic `DocumentId`).** The retained source text / `EditScript` (the backing store below) is **part of the same document revision as the graph** ŌĆö not a parallel, separately-versioned artifact. A commit advances the one `DocumentId = (UUID, revision)` and both the graph view (`rise://scene/graph`) and the text view (`rise://scene/text`) read that same revision. So the `documentId` a text read returns **describes graph+text atomically**: a `baseEpoch` computed from `rise://scene/text` is valid against a graph-mutating tool and vice-versa, and there is no window where text and graph disagree about the revision (the same torn-read closure E2 makes for the epoch, TRANSACTION_MODEL ┬¦3.4).

**Backing store** (the accessor `rise://scene/text` reads from), two options in preference order:

1. **Buffer-on-parse (preferred).** Extend `AsciiSceneParser` to retain the fully-read pre-tokenization text and expose it (`GetSourceText() const`), stash it on the `Job`/controller at load. Cheap (one string), and it's the exact substrate round-trip save already wants. The "current" text = original buffer with the round-trip `EditScript` applied (the save engine already computes that diff for [../ROUND_TRIP_SAVE_PLAN.md](../ROUND_TRIP_SAVE_PLAN.md)). The length-preserving redactor (above) runs over this buffer at read time, returning masked `text` + `redactions[]`.
2. **Re-read from disk + apply edits.** If buffering is rejected, re-read the loaded path and apply the in-memory `EditScript`. Fragile if the file changed underneath; option 1 is strictly better.

Either way the accessor is also useful to the GUI's "show me the code" panel (the un-redacted in-process view), so it's not MCP-only scaffolding. **Alternative considered and rejected:** keep `text/plain` and put the epoch in a separate resource ŌĆö rejected because a client could then read text at revision N and an epoch at revision N+1 (the torn-read window E2 closes by making the identity *intrinsic* to the payload).

### 3.2 Framebuffer encoding for a vision model (the seeŌåÆadjustŌåÆrender loop)

Returning the rendered image to a vision model is, with `validate`, one of the two things that separate a toy from something that "fully operates" the engine (roadmap ┬¦9.3). Design points:

- **Default = tone-mapped sRGB PNG.** A vision model should see *the displayed image*, not raw linear HDR (which looks flat/wrong decoded as sRGB). The default `rise://framebuffer` runs the same display transform the GUI uses (`FrameStore::Render` with an `RGBA8_sRGB` target and the active exposure/view transform). This is what makes "converge on a believable sunset through the window" judgeable ŌĆö a spectral renderer's tonal/chromatic correctness is impossible to fake in RGB, and the model can only critique what it can see correctly.
- **Query parameters** on the URI (parsed by the resource provider): `?format=png|jpeg|exr` (EXR via `FileRasterizerOutput` for an external client that wants ground-truth linear), `?exposure=<EV>` (let the model re-expose to inspect shadow/highlight detail without re-rendering), `?max_dim=<px>` (downscale to keep base64 payload + vision tokens bounded ŌĆö default cap ~1024px long edge), `?region=x,y,w,h` (crop).
- **Binary return.** Per spec, image resources return `{ uri, mimeType, blob: "<base64>" }`. Tools that render can *also* embed the image directly in their `tools/call` result as an `image` content block (`{ type: "image", data, mimeType }`) so the model sees the result in the same turn it asked to render ŌĆö no second `resources/read` round-trip. (┬¦4.5)
- **AOVs** (`rise://framebuffer/aov/{aov}`) are visualized, not raw: normals remapped `[-1,1]ŌåÆ[0,1]`, depth normalized to the frame's near/far, objectId false-colored. Raw float AOVs are available via `?format=exr` for an external tool that wants the actual data.
- **Cost discipline.** The framebuffer resource never *triggers* a render; it returns the latest completed frame (or "no frame yet"). Rendering is an explicit tool call (┬¦4.5) gated by the render scope (┬¦7), because renders consume all cores ([../../CLAUDE.md](../../CLAUDE.md) memory: never run renders concurrently). **The single-render invariant is enforced once, in the `RenderCoordinator`** ([RENDER_COORDINATOR.md](RENDER_COORDINATOR.md) ┬¦2.1 I1), not re-derived per tool ŌĆö a render tool submits a job and the coordinator decides preempt/queue/reject (┬¦4.4). A vision-loop "render this preview" maps to an isolated coordinator job against a private film (RENDER_COORDINATOR ┬¦5), so it never tears the live framebuffer a subscribing client is reading.

### 4.0 Two tool contracts ŌĆö *committed* vs *proposed* (resolves the ┬¦2.2-vs-┬¦4-handlers authority contradiction; GUI_ROADMAP ┬¦13a #3)

**Finding (external-client authority contradiction).** Two parts of this doc disagreed about who applies a mutation. ┬¦2.2 (and ┬¦2.3's coexistence model) say an **external** MCP client is **read + propose-only** ŌĆö it stages a `Propose` the owner applies. But the ┬¦4.1ŌĆō┬¦4.5 handler tables describe handlers that **commit through `SceneEditController::Commit`**, and ┬¦8 has *every* successful mutating tool return `newEpoch`/`undoAvailable` ŌĆö i.e. as if it had applied. Those two cannot both be the contract for the same caller. **Resolution: a mutating tool resolves to one of two distinct contracts at dispatch time, keyed on the session's authority, and they return different shapes.**

| Contract | Who gets it | What the handler does | What it returns |
|---|---|---|---|
| **Committed tool** | the **document owner** ŌĆö the in-app GUI session and the in-process agent-policy session (the single writer, TRANSACTION_MODEL ┬¦0 #4) | builds a `SceneTransaction(baseDocument = baseEpoch)` and calls `SceneEditController::Commit` directly (the ┬¦4.1ŌĆō┬¦4.5 tables describe *this* path) | on success, the ┬¦8 **`{ changed, newEpoch, undoAvailable }`**; on a lost race, the ┬¦8 `conflict` |
| **Proposed tool** | an **external** MCP client (loopback-HTTP), and any session holding less than commit authority | wraps the same edit set in a one-shot `Propose` (TRANSACTION_MODEL ┬¦12) that the owner applies/rejects out-of-band; it does **not** call `Commit` | a **proposal acknowledgement ŌĆö `{ proposalId, status: "pending"\|"applied"\|"rejected"\|"conflict", baseEpoch }`** ŌĆö **never** `newEpoch`/`undoAvailable` at call time (the model learns the outcome when the owner acts on the proposal, surfaced via the proposal's status / a `notifications/resources/updated` on `rise://scene/*`) |

The *committed* and *proposed* contracts wrap the **same** validated edit set and obey the **same** `baseEpoch` precondition (┬¦4 head) and the **same** ┬¦4.6 `validate`-first rule; they differ only in whether the handler commits or stages, and therefore in the return shape. The per-tool sketches in ┬¦4.1ŌĆō┬¦4.5 and the ┬¦8 success return describe the **committed** contract; for a proposed-contract caller the same row applies with "commit" read as "stage a `Propose`" and the ┬¦8 success return replaced by the proposal acknowledgement above. The server picks the contract from the session's authority ŌĆö the model never chooses it.

**Headless stdio is external but has no GUI owner ŌĆö the headless autonomy policy (`rise --agent-stdio`).** `AgentStdioTransport` (┬¦2.3) is an *external* process boundary (so by the table above it would be propose-only) **but there is no in-app/GUI session to approve a proposal** ŌĆö a `Propose` with no owner would hang forever. A headless server therefore runs under an explicit, operator-set **headless autonomy policy**, never an implicit default:

- **Default ŌĆö `read+validate` only.** Absent an explicit grant, a headless stdio server exposes only the `read` scope (resources + `validate` + `list_*`); every mutating `tools/call` is refused with a structured error directing the caller to start the server with an autonomy grant. This is the safe default for "RISE as an MCP server a script spawned" ŌĆö it cannot mutate without the operator opting in.
- **Operator-set auto-apply (opt-in, the headless owner).** A launch flag / config (e.g. `--agent-autonomy=commit`, set by the human who spawned the process ŌĆö never by the model, a scene file, or an external server, mirroring the ┬¦4.4 endpoint-immutability rule) authorizes the **committed** contract on that headless server: the operator's act of launching with the grant *is* the owner approval, so mutating tools commit directly and return `newEpoch`/`undoAvailable`. The OS process boundary is the trust boundary (AI_SECURITY_MODEL ┬¦8.1.6); the always-confirm set (┬¦4.4 / AI_SECURITY_MODEL ┬¦5) is the one carve-out ŌĆö its ops (`remove_entity`, `load_scene`, overwrite save/export, enabling an external MCP) still require a confirmation the model cannot satisfy, so a headless `--agent-autonomy=commit` does **not** silently auto-run them (it fails them closed unless a separate, explicit headless-destructive grant is given). The matching LLM-runtime view is LLM_AGENT_RUNTIME ┬¦5.x / ┬¦6 (autonomy levels); this is the transport-surface statement of it.

(In-process is the third topology: it is the owner by construction ŌĆö shared address space, the in-app session ŌĆö and always gets the *committed* contract. So the contract map is: in-process ŌåÆ committed; loopback-HTTP external ŌåÆ proposed; stdio headless ŌåÆ read+validate by default, committed only under the operator's autonomy grant.)

Tools are model-invoked. Every tool carries MCP **annotations** (`readOnlyHint`, `destructiveHint`, `idempotentHint`, `openWorldHint`) so a host can gate them (the spec says clients SHOULD keep a human in the loop and MUST treat annotations from untrusted servers as untrusted ŌĆö but RISE is a *trusted first-party* server, so its annotations are authoritative for our own in-app client). **Annotations are advisory; the server-side scope check (┬¦7) is the gate, and the security policy is owned by [AI_SECURITY_MODEL.md](AI_SECURITY_MODEL.md).** Each tool's `inputSchema` is JSON Schema 2020-12 ŌĆö a first-pass filter validated by the conformance test (┬¦5, VALIDATION_ARCHITECTURE ┬¦5.4), with `validate` (┬¦4.6) as the authority. **Every tool taking a file path resolves it through the `ProjectRootJail`** (AI_SECURITY_MODEL ┬¦4) ŌĆö no model-supplied absolute or `..` paths; Save-As to a new location is materialized only through the human OS file dialog, never a model-chosen string.

**`baseEpoch` is a required field on every mutating tool's `inputSchema` (the precondition, second-review ┬¦2).** The ┬¦2.2 prose mandates the precondition but the v1 schemas below did not carry it ŌĆö so the model had nowhere to *put* the revision it read, and the precondition was unenforceable from the protocol surface. Every mutating tool now declares:

```jsonc
"baseEpoch": {                       // the DocumentId precondition (TRANSACTION_MODEL ┬¦3.4)
  "type": "object",
  "required": ["uuidHi","uuidLo","revision"],
  "properties": {
    "uuidHi":   { "type": "integer" },   // the document UUID hi/lo from rise://scene/* (┬¦3.1)
    "uuidLo":   { "type": "integer" },
    "revision": { "type": "integer" }    // the per-commit revision observed on that read
  },
  "description": "The (UUID, revision) read from rise://scene/text|graph. The commit is accepted only if it still matches the live document; a UUID mismatch means the doc was reloaded, a revision mismatch means it advanced."
}
```

`baseEpoch` is named for the ┬¦2.2 prose and the runtime/security siblings, but it carries the **whole `DocumentId` pair** (TRANSACTION_MODEL ┬¦3.4 closes the reload-coincidence hole by matching the UUID too, not the revision alone ŌĆö a bare revision could falsely match a freshly-reloaded document). Each mutating handler builds a `SceneTransaction` whose `baseDocument = baseEpoch` and commits through `SceneEditController::Commit`; on a lost race it returns the `conflict` error contract of TRANSACTION_MODEL ┬¦4.4 carrying the live `(UUID, revision)` (re-read `rise://scene/*`, retry) ŌĆö never a silent clobber. To reduce churn the per-tool sketches below write `baseEpoch` as a leaf and refer here for its shape.

**Parallel-tool semantics (no silent last-write-wins, second-review ┬¦2).** Providers can emit several `tools/call`s in one assistant turn (Claude parallel `tool_use`, Gemini parallel `functionCall`). If two mutating calls in one batch share the **same** `baseEpoch` they read, the **first to reach `Commit` advances the revision, making every sibling's `baseEpoch` stale.** RISE does **not** silently apply the rest on top (that would be last-write-wins, the exact failure TRANSACTION_MODEL exists to prevent). Two sanctioned shapes, both at the dispatcher (not per-tool):

1. **Reject-stale-then-reread (default).** The dispatcher commits batched mutating calls **one at a time in arrival order**; the first succeeds and bumps the revision; each subsequent call whose `baseEpoch` no longer matches returns `conflict` with the new live `(UUID, revision)`. The agent runtime feeds the conflict back so the model re-reads `rise://scene/*` and re-issues the remaining edits against the fresh base (LLM_AGENT_RUNTIME ┬¦5.5 / ┬¦7.4). The model is told in the tool description that parallel mutations on one base are serialized and later ones may conflict.
2. **Batch-as-one-transaction (opt-in).** When the model's intent is "apply all of these atomically," the host composes the batch into a **single** `SceneTransaction` (one `baseEpoch`, all edits, one commit, one undo unit) via the per-client transaction builder / one-shot `Propose` (TRANSACTION_MODEL ┬¦12). Either the whole batch applies at the read revision or the whole batch conflicts ŌĆö never a partial last-write-wins.

Read-only tools in the same batch are unaffected (they carry no `baseEpoch` and read a published snapshot, TRANSACTION_MODEL ┬¦6.2).

**Persistence ŌĆö corrected against [CURRENT_STATE_AUDIT.md](CURRENT_STATE_AUDIT.md) (round-trip save is LARGELY IMPLEMENTED, not Phase-0-future).** `SaveEngine` ships a two-mode engine **plus Phase B property re-emit and Phase C created-entity emit** (`SaveEngine.cpp`, 1,898 lines; `tests/SaveEngineTest.cpp`, 41 tests; CURRENT_STATE_AUDIT ┬¦1). The legend below reflects what actually persists *today*:
- **today (structured)** ŌĆö persists *now* as a structured save (in-place byte splice or managed-override block) through `SceneEditController` ŌåÆ `SaveEngine`, not via wholesale rewrite. Covers **transforms, non-transform property re-emit (Phase B: camera/light/material/medium + object bindings), and created-camera emit (Phase C)**.
- **today (wholesale)** ŌĆö works now but persists only via the wholesale rewrite-and-reload path (`validate` ŌåÆ `apply_scene_text` ŌåÆ reload, ┬¦4.4; roadmap ┬¦9.5). The cheap fallback for edits the structured save engine can't yet represent.
- **genuine gap: non-camera creation** ŌĆö Phase C re-emits **cameras only** (`SaveEngine.cpp:1351`, "V1: only cameras are creatable"; a non-camera chunk in the managed block is a hard **Refused**, `:1373`). **Render-control tools carry no `baseEpoch`** (excluded from the ┬¦4-head precondition rule): `render`, `stop_render`/`pause_render`/`resume_render`, `set_region`/`clear_region` do **not** mutate the scene graph ŌĆö they allocate compute or touch render-state. They read the currently-published snapshot (a render is a coordinator job against a snapshot, TRANSACTION_MODEL ┬¦6.4); a render computed against revision N while N+k is live is reconciled as *stale* by the `RenderCoordinator` (RENDER_COORDINATOR), not rejected by a precondition. Tools that produce a `SceneTransaction` (┬¦4.1ŌĆō┬¦4.3, ┬¦4.5 `apply_scene_text`) take the full `baseEpoch`.

**Save / load / export are NOT graph-mutating, but they are NOT precondition-free either (resolves the ┬¦4.4 over-broad exemption; GUI_ROADMAP ┬¦13a #7).** The earlier rule lumped `save_scene`/`load_scene`/`export_*` in with the render tools as "no `baseEpoch` because they don't mutate the graph." That is wrong on two counts, and each needs a *document-identity* precondition (an **expected `DocumentId`**, not the full mutating-tool `baseEpoch` ŌĆö these don't commit a transaction, they pin which document/revision the operation must apply to):

- **`load_scene` *replaces the document*** (it re-roots the session and swaps the controller ŌĆö a document swap, ┬¦4.5 / TRANSACTION_MODEL ┬¦7.5). It is at least as consequential as a mutation and is already in the always-confirm set (┬¦4.4 / AI_SECURITY_MODEL ┬¦5). It carries an **expected current `DocumentId`** so a load issued from a stale view (the doc was reloaded/swapped out from under the model since it last read) is rejected with `conflict` rather than silently discarding a now-different document the user is working in.
- **`save_scene` checkpoints a *specific revision*.** Saving must persist the revision the model *intended*, not whatever happens to be live when the write lands. `save_scene` takes an **expected `DocumentId`** and acquires a **consistent save lease**: the engine pins the document at that revision for the duration of the save (no commit interleaves mid-write), writes it, and **returns the saved `DocumentId` (the `{ uuidHi, uuidLo, revision }` actually persisted)** in its structured result. If the live revision has advanced past the expected one when the lease is requested, the call returns `conflict` with the live `DocumentId` (re-read and retry) ŌĆö never a torn save that mixes two revisions.
- **`export_image`/`export_movie` identify the snapshot/frame they encoded.** An export is a read of a rendered frame, so it pins (and **returns**) the **frame `generation`** it encoded (`FrameStore::Generation()`, MCP ┬¦11 open-q 3 / `rise://render/stats`) and the **`DocumentId` of the scene that produced that frame**, so the model can tell *which* document+frame the file corresponds to and never mistakes a stale framebuffer for the current one. An export computed against a frame that has since been superseded still succeeds (it faithfully encodes the frame it had) but the returned `generation`/`DocumentId` make the staleness observable.

So the precondition map is: render-control ŌåÆ none (coordinator reconciles staleness); `save_scene`/`load_scene`/`export_*` ŌåÆ an **expected `DocumentId`** (+ a save lease for `save_scene`; + a returned frame `generation` for exports); graph-mutating tools ŌåÆ the full `baseEpoch` transaction precondition.

## 5. Schema auto-generation (a first-pass filter, kept honest by a conformance test)

Tool and resource schemas are generated from `SceneGrammar::Describe()` ŌĆö the same source that keeps the syntax highlighter and the autocomplete engine honest ([../../src/Library/Parsers/README.md](../../src/Library/Parsers/README.md)). **Correction (retracts the earlier "can never drift" / "schema = exact parser" claim).** An adversarial review found a *generated* schema is **not** the parser and can be **stricter** than it ([VALIDATION_ARCHITECTURE.md](VALIDATION_ARCHITECTURE.md) ┬¦5): the generic dispatcher only checks that numeric kinds are *finite numeric tokens* (`AsciiSceneParser.cpp:725-742`) ŌĆö it does **not** enforce vector cardinality, integer-ness, or enum membership, and some declared enum lists are intentionally *suggestions* (e.g. `pathguiding_sampling_type` accepts any string). A schema that emits `minItems/maxItems`, `type:integer`, or a closed `enum` for those would over-reject values the parser honors. So the schema is a **fast first-pass filter** for the LLM and editor, **held honest by `tests/SchemaConformanceTest.cpp`** (VALIDATION_ARCHITECTURE ┬¦5.4) which fails on any over- or under-acceptance versus the real parser; the **`validate` tool (┬¦4.6/┬¦6) is the authority**, because only it runs the parser's full accept/reject (including the runtime-manager checks ┬¦6 / VALIDATION_ARCHITECTURE ┬¦1.4 that no static schema can express).

### 5.1 `ValueKind` ŌåÆ JSON Schema mapping

`SchemaGen` maps each `ParameterDescriptor` to a JSON Schema fragment. **The richer constraints below (closed-vs-open enum, cardinality, integer-ness, the `color`/`scalar` pipe) ride on the adopted `pipe` field on `ParameterDescriptor`** ([../GUI_ROADMAP.md](../GUI_ROADMAP.md) ┬¦16; owned by [MATERIAL_EDITOR.md](MATERIAL_EDITOR.md)) and the conformance test gates them:

| `ValueKind` | JSON Schema |
|---|---|
| `Bool` | `{ "type": "boolean" }` |
| `UInt` | `{ "type": "integer", "minimum": 0 }` |
| `Double` | `{ "type": "number" }` |
| `DoubleVec3` | `{ "type": "array", "items": {"type":"number"}, "minItems":3, "maxItems":3 }` |
| `DoubleVec4` | `ŌĆ”minItems/maxItems:4` |
| `DoubleMat4` | `ŌĆ”minItems/maxItems:16` |
| `String` | `{ "type": "string" }` |
| `Filename` | `{ "type": "string", "description": "path resolved via RISE_MEDIA_PATH, confined by the ProjectRootJail" }` (no absolute/`..` paths from the model ŌĆö AI_SECURITY_MODEL ┬¦4) |
| `Enum` (**closed**) | `{ "type": "string", "enum": [ ŌĆ”enumValues ] }` |
| `Enum` (**open/suggestion**) | `{ "type": "string", "examples": [ ŌĆ”enumValues ] }` ŌĆö **not** `enum`; the parser accepts any string (VALIDATION_ARCHITECTURE ┬¦5.2). Closed-vs-open is carried on the `pipe`/richer-constraint field, **not** inferable from `enumValues` alone |
| `Reference` | `{ "type": "string", "description": "name of an existing <categories> chunk" }` (and, for the in-app client, an `x-rise-reference-categories` hint the host can use to offer completions). Existence is **not** a schema check ŌĆö it is a `full`-mode `validate` check (┬¦6) |

Carried through verbatim into the schema: `description` ŌåÆ `description`, `required` ŌåÆ the chunk schema's `required` array, `defaultValueHint` ŌåÆ `default` (string form), `unitLabel` ŌåÆ appended to `description` and emitted as `x-rise-unit`, `presets` ŌåÆ `examples` (and `x-rise-presets`), `tupleKinds` ŌåÆ a `prefixItems` tuple schema, `repeatable` ŌåÆ wrap as `{ "type":"array", "items": <param schema> }`.

### 5.2 Where the generated schema is used

1. **The grammar resource** `rise://grammar/schema` = `{ "chunks": { "<keyword>": { "category", "description", "parameters": { "<name>": <fragment> }, "required": [...] }, ŌĆ” } }`, one entry per `SceneGrammar::AllChunkKeywords()`. This is the AI's authoritative "what is valid" reference.
2. **Per-entity tool `inputSchema`s.** `set_property`'s `value` and `apply_material`'s `material_chunk` don't enumerate every possibility inline; instead the model is steered to read `rise://grammar/chunk/{keyword}` first. But `add_entity` / `create_camera` / `set_lens` *can* publish tighter schemas by composing the relevant chunk fragment, so the model gets inline validation hints for the common cases.
3. **`validate`'s rule set.** `SceneValidator` (┬¦6) runs the **real parser** against an isolated throwaway `Job` (VALIDATION_ARCHITECTURE ┬¦3), so its verdict matches `ParseAndLoadScene` *in practice* ŌĆö including the runtime-manager checks (┬¦6 / VALIDATION_ARCHITECTURE ┬¦1.4) the schema cannot express. The schema is the cheap pre-filter; `validate` is the oracle.

### 5.3 Why this matters

When someone adds a parameter to a chunk, they edit exactly one `Describe()` entry and one `Finalize()` read ([../../src/Library/Parsers/README.md](../../src/Library/Parsers/README.md)). That single edit updates the parser, the highlighter, the autocomplete, the grammar resource, the affected tool schemas, *and* `validate` from one source ŌĆö but **"updates from one source" is not "cannot drift."** The generated schema can still over- or under-accept versus the parser (┬¦5 head; VALIDATION_ARCHITECTURE ┬¦5.1ŌĆō┬¦5.2). The property is *earned*, not asserted: `tests/SchemaConformanceTest.cpp` (VALIDATION_ARCHITECTURE ┬¦5.4) makes drift a **red test** by checking `schema.accepts(case) == parser.accepts(case)` over a per-chunk corpus, and a coverage guard asserts `SchemaGen` emits one entry per `SceneGrammar::AllChunkKeywords()` so a new chunk can't silently fall out of the AI's view.

## 6. `validate` ŌĆö the dry-run parse (design owned by VALIDATION_ARCHITECTURE)

`SceneValidator::Validate(const std::string& sceneText, mode) -> ValidationReport` parses scene text **with no side effects** (no live-`Job` mutation, no rendering, no file writes) and returns structured diagnostics an LLM can self-correct from. **The full corrected design is [VALIDATION_ARCHITECTURE.md](VALIDATION_ARCHITECTURE.md); this section is a summary that defers to it.**

### 6.1 Implementation strategy (corrected ŌĆö the capturing-mock approach is rejected)

The original sketch here proposed a **capturing `ValidationJob` mock** that is simultaneously side-effect-free *and* parser-identical. An adversarial review found that is **not buildable as a small, faithful component** (VALIDATION_ARCHITECTURE ┬¦1): the parser **builds as it parses** (every `Finalize` calls `pJob.Add*`, `AsciiSceneParser.cpp:9862`), so "parser-identical" requires running `Finalize`, and running `Finalize` *is* the side effect; the decisive checks (e.g. the IScalarPainter pipe decision in `Job::ResolveOrDiagnoseScalar`, `Job.cpp:2718`) live in `Job`, not the descriptor, so a faithful mock would have to reimplement a large fraction of `Job`'s ~217 pure-virtual surface ŌĆö a second source of truth that *will* drift. The corrected mechanism:

1. **String input.** Add a string-input entry (`ParseString`) to `AsciiSceneParser` so the validator needs no temp file. Independently useful (tests, the editor gutter).
2. **Isolated *real* throwaway `Job` (not a mock).** Run the normal descriptor dispatch + `Finalize` against a **fresh, private `Job`** that shares no state with the live document. Because `Finalize` mutates only the throwaway (and its isolated managers), the full semantic pass runs and the verdict matches the real parser ŌĆö and the only mutations are to an object discarded at end of call (VALIDATION_ARCHITECTURE ┬¦3). The end-state separates this cleanly via `parse ŌåÆ IR ŌåÆ validate IR ŌåÆ apply IR` (VALIDATION_ARCHITECTURE ┬¦4).
3. **Command policy.** Validation does **not** route `>`-commands through `AsciiCommandParser` unmodified ŌĆö a barrier shim refuses `quit`/`render`/`load`/`run`/`mediapath`/`clearall`/`remove` so `exit(1)` (`AsciiCommandParser.cpp:182`) is structurally unreachable from a validate call (VALIDATION_ARCHITECTURE ┬¦2). This is the strict end of the **shared** destructive-command taxonomy [AI_SECURITY_MODEL.md](AI_SECURITY_MODEL.md) ┬¦5 also consumes.
4. **Diagnostics with location.** Reuse the round-trip-save byte-range token spans ([../ROUND_TRIP_SAVE_PLAN.md](../ROUND_TRIP_SAVE_PLAN.md); `SourceSpanIndex` `RawTokenSpan`) for line/column; fall back to chunk-keyword + parameter-name context where a span isn't attachable.
5. **Two depths** (`mode`, default `full`): `syntax` = chunk/param/type validity only (fast, the editor gutter path); `full` = also resolve references and check declared filenames resolve under `RISE_MEDIA_PATH`. `full` is what the AI should call before committing. Cross-file `> load` / `> mediapath` semantics are deliberately out of scope for a single-text validation (VALIDATION_ARCHITECTURE ┬¦3.4).

### 6.2 `ValidationReport` shape (returned as MCP structured content)

```json
{
  "ok": false,
  "diagnostics": [
    {
      "severity": "error",
      "code": "UNKNOWN_PARAMETER",
      "chunk": "ggx_material",
      "parameter": "roughnes",
      "line": 42, "column": 5,
      "message": "Parameter 'roughnes' is not declared on chunk 'ggx_material'.",
      "suggestion": "roughness"
    },
    {
      "severity": "error",
      "code": "UNRESOLVED_REFERENCE",
      "chunk": "standard_object",
      "parameter": "material",
      "line": 88,

```

Diagnostic `code`s (stable, machine-matchable; **canonical set in VALIDATION_ARCHITECTURE ┬¦6.2**): `SYNTAX_ERROR`, `SCENE_VERSION`, `UNKNOWN_CHUNK`, `UNKNOWN_PARAMETER`, `TYPE_MISMATCH`, `VECTOR_CARDINALITY` (a `DoubleVec3` that got the wrong token count), `NON_INTEGER_UINT`, `ENUM_VALUE_INVALID` (**closed** enums only), `MISSING_REQUIRED`, `DUPLICATE_NAME`, `UNRESOLVED_REFERENCE`, `WRONG_PIPE` (the `ResolveOrDiagnoseScalar` color-vs-scalar failure), `FILE_NOT_FOUND`, `BARRIER_COMMAND` (a `>` barrier command per ┬¦6.1.3). The `suggestion`/`candidates` fields reuse the existing `SuggestionEngine` ranking (nearest known parameter / nearest existing reference of the right category) so the model gets the fix, not just the complaint.

| **edit** | read + scene-mutation/structural/camera tools (┬¦4.1ŌĆō4.3) + `apply_scene_text` + `save_scene` | `render`, `export_*`, `load_scene` | L1 Propose-and-confirm / L2 within edit |
| **render-and-spend** | edit + `render` / region / `export_*` (consumes compute, may cost cloud tokens upstream) | ŌĆö | L2 Operate-with-guardrails / L3 |

Cross-cutting, non-negotiable (full design in [AI_SECURITY_MODEL.md](AI_SECURITY_MODEL.md)):

- **Loopback only.** The HTTP transport binds `127.0.0.1`, validates `Origin` (403 on mismatch, per spec DNS-rebinding guidance), and requires a per-launch token. No `0.0.0.0`, ever (AI_SECURITY_MODEL ┬¦8; same-LAN is forbidden, incl. Android ŌĆö adb-loopback only, ┬¦8.2).
- **No shell, no arbitrary FS ŌĆö path-root jailed.** There is **no** "run command" tool and no general file-read/write tool. **Every file-path tool argument resolves through the `ProjectRootJail`** (AI_SECURITY_MODEL ┬¦4): no model-supplied absolute or `..` paths; canonicalize-then-confine to the project root + media/asset roots (reads) / project root (writes). Writing elsewhere (Save-As / Export) is materialized **only** through the human OS file dialog, never a model-chosen string. `openWorldHint:true` still flags the FS-touching tools so a host can prompt.
- **Render arbitration is delegated, not a flat busy-bit.** The single-render invariant lives in the `RenderCoordinator` ([RENDER_COORDINATOR.md](RENDER_COORDINATOR.md) ┬¦2.1 I1); a render tool submits a job and the coordinator decides preempt/queue/reject (┬¦4.4). A host MAY still cap renders-per-minute for L3 loops (roadmap ┬¦9.4).
- **Destructive / off-machine ops are always-confirm.** `remove_entity`, `load_scene`, file *overwrite* via `save_scene`/`export_*`, enabling an external MCP server, and the first cloud egress of new content require a human confirmation **the model cannot satisfy**, at every autonomy level incl. L3 (AI_SECURITY_MODEL ┬¦5). `destructiveHint:true` is the advisory hint; the always-confirm gate is the enforcement.
- **Everything reversible is undoable, attributed & diffable.** Because all mutations route through `SceneEditController::Commit` (principle #6; TRANSACTION_MODEL ┬¦4), every AI action lands in the one undo stack **attributed to its session** (TRANSACTION_MODEL ┬¦8) and is visible in the "show me the code" panel ŌĆö the diff-review gate that makes L1+ safe. An injected "delete all lights" surfaces as a red diff the user rejects (AI_SECURITY_MODEL ┬¦2.2.2).

## 8. Error & diagnostic format (so the LLM can self-correct)

MCP distinguishes two error channels ([spec/server/tools](https://modelcontextprotocol.io/specification/2025-11-25/server/tools)), and we use them deliberately:

- **Protocol errors** (JSON-RPC `error`, e.g. `-32602`) ŌĆö for malformed requests the model is unlikely to fix: unknown tool name, arguments that don't satisfy `inputSchema`, unknown resource URI (`-32002`). The model rarely recovers from these, so we keep them rare by making schemas precise.
- **Tool execution errors** (`{ content: [...], isError: true }`) ŌĆö for actionable failures the model *should* retry differently: a `set_property` value that fails descriptor validation, an `add_entity` whose `chunk_text` fails `validate`, a `save_scene` that returns `Refused`, an `export_image` whose path the `ProjectRootJail` refuses (`PATH_OUTSIDE_PROJECT_ROOT`, AI_SECURITY_MODEL ┬¦4), or a mutating tool whose `baseEpoch` lost the race (a **`conflict`** carrying `currentEpoch: { uuidHi, uuidLo, revision }` ŌĆö the live `DocumentId`, TRANSACTION_MODEL ┬¦4.4; a UUID mismatch means the document was *reloaded* and the model must re-bootstrap, a revision mismatch means it advanced and the model rebases ŌĆö telling the model to re-read `rise://scene/*` and retry). These carry the **same `ValidationReport`/diagnostic JSON as ┬¦6** in `structuredContent`, plus a human-readable text block. This is the spec's intended self-correction path ("Tool Execution Errors contain actionable feedback that language models can use to self-correct").

Concretely, a mutation tool that touches scene text **always runs `validate` first** and, on failure, returns `isError:true` with the diagnostics rather than mutating ŌĆö so the model gets line/parameter/suggestion feedback and tries again, never leaving the scene half-edited. **The success return depends on which ┬¦4.0 contract the caller got.** A **committed-contract** call (the owner/in-app session, or a headless server under an `--agent-autonomy=commit` grant) returns structured content describing what changed (`{ changed: [...], newEpoch: { uuidHi, uuidLo, revision }, undoAvailable: true }` ŌĆö `newEpoch` is the post-commit `DocumentId` from `CommitResult.newDocument`, TRANSACTION_MODEL ┬¦4.1) so the model can confirm, carry it as the `baseEpoch` of its next mutation, and the host can refresh. A **proposed-contract** call (an external MCP client; ┬¦4.0) returns instead a proposal acknowledgement (`{ proposalId, status: "pending"|"applied"|"rejected"|"conflict", baseEpoch: { uuidHi, uuidLo, revision } }`) ŌĆö it carries **no** `newEpoch`/`undoAvailable`, because nothing has committed yet; the model learns the post-apply `DocumentId` only once the owner acts on the proposal (surfaced via the proposal's resolved `status` and a `notifications/resources/updated` on `rise://scene/*`), and must re-read `rise://scene/*` to pick up the new base before its next edit.

## 10. Non-goals / deliberately NOT doing

- **No second mutation path.** The server never edits `Job`/`Scene` directly; every mutation goes through `SceneEditController::Commit` (principle #6; TRANSACTION_MODEL ┬¦4). A tool that "just sets a field on the object" bypassing the controller is forbidden ŌĆö it would break undo, the dirty tracker, and the diff panel.
- **No platform-specific server.** There is exactly one `AgentServer` in shared C++ (`src/Library/Agent/`). No Obj-C / Swift / Qt / Kotlin reimplementation. (Principle #2.)
- **No shell / arbitrary-filesystem / network-egress tool.** RISE's agent server operates *RISE*, not the machine ŌĆö file paths are `ProjectRootJail`-confined and outbound URLs pass `IOutboundUrlPolicy` (┬¦7; AI_SECURITY_MODEL ┬¦4/┬¦3).
- **No hand-maintained tool/resource schema.** Schemas are generated from `SceneGrammar` (┬¦5) and held honest by the conformance test ŌĆö but the generated schema is a **first-pass filter, not the parser** (`validate` is the authority; ┬¦5, VALIDATION_ARCHITECTURE ┬¦5).
- **No binding to non-loopback interfaces.** Even the HTTP transport is 127.0.0.1 + token + Origin-checked. Remote operation (and same-LAN, incl. Android) is explicitly out of scope ŌĆö a remote-render service is a different product (AI_SECURITY_MODEL ┬¦8, ┬¦11).
- **No second render arbiter, and no concurrent renders.** Renders never run concurrently (cores are fully consumed per render ŌĆö [../../CLAUDE.md](../../CLAUDE.md)); the server does not invent its own scheduling. Admission (queue / preempt / coalesce / reject-if-busy) is the single `RenderCoordinator`'s job ([RENDER_COORDINATOR.md](RENDER_COORDINATOR.md)) ŌĆö this doc's earlier "flat reject-if-busy, no queue" rule is superseded by the coordinator's per-class policy (a queue *does* exist there for thumbnails/previews; full-scene renders still serialize one-at-a-time).
- **No replacing the GUI panels.** The descriptor-driven properties panel and the editor stay; the agent server is a parallel authoring surface over the same controller, not a substitute UI.

## 11. Open questions / flagged uncertainties

1. **JSON-RPC codec choice.** Reuse RISE's existing glTF JSON parser, or vendor a small header-only JSON-RPC library? The glTF parser may be too schema-specific for arbitrary JSON-RPC; resolve during M0. (Implementation detail, not architectural.)
2. **Thread-affinity for the loopback/stdio command queue.** The controller's "edits on the owner thread" contract is clear for the in-app client (post to run loop). For a *headless* stdio server there is no run loop ŌĆö confirm the simplest correct model is "the stdio read loop *is* the owner thread, and renders run on the controller's render thread as today." Likely fine; verify against `SceneEditController` start/stop semantics.
3. **Framebuffer freshness vs. generation counter.** `FrameStore::Generation()` gives a monotonic dirty-check; confirm `rise://framebuffer` should return the latest *completed* frame and expose `generation` in `rise://render/stats` so a subscribing client can tell "is this the frame I asked for?" (Avoids returning a half-accumulated preview to a vision model.)
4. **`validate` on partial chunks.** `{ chunk_text }` needs wrapping in a minimal valid header (`RISE ASCII SCENE 6` + a `film` chunk?) to parse standalone, or a dispatcher entry point that validates one chunk against its descriptor without a full scene. The latter is cleaner and reuses `SceneGrammar::FindChunk` + the dispatch validation directly ŌĆö prefer it.
5. **Per-session scene isolation for stdio batch.** If one headless process should serve *several* scenes (CI rendering a directory), does each `load_scene` reset the single controller, or do we allow multiple documents per server? M0 assumes one document per server (simplest); multi-document is an M3+ consideration.
6. **Variance/RMSE availability ŌĆö and it must be built first.** `rise://render/variance` has **no backing accessor in master** (the per-pixel `ProbeResult` is private, `AutoRasterizer.h:192`; CURRENT_STATE_AUDIT ┬¦11). Before this resource can ship it needs the net-new additive read-back from SPECTRAL_DIFFERENTIATORS D5. Even then it is only meaningful when adaptive sampling or a reference image exists; define the "not available" response shape so the model doesn't treat absence as zero variance. (Pairs with the Žā┬▓┬ĘT "rewards dark images" caveat in [../../CLAUDE.md](../../CLAUDE.md) ŌĆö never report variance without the RMSE companion when a reference is present.)

## 12. Acceptance criteria (per [GUI_ROADMAP.md](../GUI_ROADMAP.md) ┬¦15)

- **Tests** ŌĆö schema-conformance (the generated schema's *error* set matches the tolerant parser, per [VALIDATION_ARCHITECTURE.md](VALIDATION_ARCHITECTURE.md) ┬¦5.4); transport security (loopback-HTTP returns 403 on bad `Origin`, requires the per-launch token); `validate` self-correction round-trip (malformed text ŌåÆ diagnostics ŌåÆ fixed ŌåÆ accepted); scope enforcement (a `read`-scope session is denied every mutating tool + `render`); **`baseEpoch` conflict** (a stale `(UUID, revision)` precondition returns `conflict`, no write); **two tool contracts (┬¦4.0)** ŌĆö an owner/in-process mutating call returns `{ changed, newEpoch, undoAvailable }` and commits; the *same* tool from an external (loopback-HTTP) session returns a `{ proposalId, status }` acknowledgement and does **not** commit until the owner approves; **headless autonomy (┬¦4.0)** ŌĆö a `rise --agent-stdio` server with no autonomy grant refuses every mutating tool (read+validate only), and with `--agent-autonomy=commit` commits directly yet still fails-closed on the always-confirm set; **single-byte mask + no-wholesale-when-redacted (┬¦3.1)** ŌĆö `rise://scene/text` masks each secret byte with a one-byte ASCII mask so `len(redacted)==len(canonical)` and all offsets are stable (a multi-byte `ŌĆó` mask is a red test), and `apply_scene_text` of text from a read that carried `redactions[]` is refused (`REDACTED_WHOLESALE_REPLACE`), not silently applied; **save/load/export document-identity (┬¦4.4)** ŌĆö `save_scene` under a save lease persists the *expected* revision and returns the saved `DocumentId`, a stale `expectedDocument` returns `conflict`; `load_scene` from a stale view is rejected; `export_*` returns the encoded frame `generation` + producing `DocumentId`; redaction round-trip (secrets never appear in `rise://*` egress, but the on-disk save stays byte-identical); framebuffer freshness (`rise://framebuffer` returns the latest *completed* frame + `generation`, never a half-accumulated preview).
- **Platform parity** ŌĆö server is shared C++ (`src/Library/Agent/`); in-process + loopback-HTTP + stdio transports on desktop; Android embeds the in-process server via JNI (adb-forwarded loopback for an external client). Identical tool/resource catalog on all platforms.
- **Performance budget** ŌĆö `validate` IR (gutter) tier interactive-fast; the deep `Job` tier debounced; read resources are O(snapshot-acquire); **no production-render regression** (the server adds nothing to the render path ŌĆö render tools enqueue `RenderCoordinator` jobs).
- **Memory budget** ŌĆö bounded per-session subscription/resource state; framebuffer encodes transient; snapshot refs released per the `RenderCoordinator` job lifecycle.
- **Accessibility** ŌĆö the protocol is headless; human-facing affordances (enable-AI toggle, token display, consent prompts) live in the chat UI ([LLM_AGENT_RUNTIME.md](LLM_AGENT_RUNTIME.md)) and follow its keyboard/focus rules.
- **Packaging** ŌĆö `rise --mcp-stdio` is a CLI entry on the existing binary; the loopback server is in-process (no new daemon); the five-build-project rule applies to new `src/Library/Agent/` files.
- **Migration** ŌĆö purely additive (no scene-format / ABI break; new C-ABI exports appended to `RISE_API.h`); existing headless `AsciiCommandParser` scripts unaffected.
- **Rollback** ŌĆö the agent server is **off by default** ("Enable AI access" opt-in); disabling it removes the surface with zero effect on rendering or the GUI.
- **Android tier** ŌĆö **Tier A** (server + chat is the mobile-native headline); loopback-HTTP external attach is Tier-C (adb-forward).
