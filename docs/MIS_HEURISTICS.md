# MIS Heuristics in RISE

Reference for **which multiple-importance-sampling heuristic each
integrator uses, why, and how that compares to state-of-the-art
renderers**.  Read this before proposing to "unify" or "fix" the
asymmetry — the asymmetry is the SOTA convention, not a historical
accident, and changing it would either break VCM's recurrence
architecture or leave variance on the table in BDPT.

## TL;DR

| Integrator | Heuristic | Hardcoded? | Configurable | Matches SOTA |
|---|---|---|---|---|
| Path tracing (BSDF vs NEE) | Power-2, optional optimal-MIS | Default | `optimalMIS` flag | ✓ (PBRT, Mitsuba, Cycles) |
| Path tracing (volumetric DT vs equiangular) | Balance (50/50) | Yes | — | ✓ |
| BDPT | Power-2 (β=2) | Effectively yes | Compile-time `MISWEIGHT_BALANCE_HEURISTIC` (off) | ✓ (PBRT-v4, Mitsuba 0.6/3) |
| VCM | Balance (β=1) | **Architectural** | — | ✓ (Georgiev 2012, SmallVCM, SmallUPBP, Mitsuba) |
| MLT | Power-2 (inherits BDPT) | Yes | — | ✓ |
| SMS | No per-strategy MIS reweight (auxiliary) | — | — | ✓ (Mitsuba 3) |

**Rule of thumb**: when adding a new connection-based integrator, use
power-2.  When adding a new merging-based integrator, use balance.
Mixing the two within one integrator is the architectural break that
Georgiev 2012's running-quantities formulation specifically avoids.

## Background — what the heuristics actually are

For a path generated by sampling strategy `s` from a set of `N`
strategies, MIS combines them via a weight `w_s(x)`:

- **Balance heuristic (β=1)**: `w_s = p_s / Σ p_i`
- **Power heuristic (β=2)**: `w_s = p_s² / Σ p_i²`

Veach 1997 §9.2 proved both are strictly better than uniform weights
or single-strategy estimators.  The balance heuristic has a provable
near-optimal upper bound on variance.  The power heuristic is an
empirical refinement that **sharpens the weight on the dominant
strategy** — outlier strategies' contributions decay quadratically
rather than linearly.  Veach recommends β=2 in practice for
connection-based MIS where each strategy is enumerated explicitly.

## Per-integrator detail

### 1. Path tracing — power-2 (default), optional optimal MIS

Code: [src/Library/Rendering/RayCaster.cpp:782-792](../src/Library/Rendering/RayCaster.cpp).

```cpp
Scalar w_bsdf;
if( rc.pOptimalMIS && rc.pOptimalMIS->IsReady() ) {
    const Scalar alpha = rc.pOptimalMIS->GetAlpha( rast.x, rast.y );
    w_bsdf = MISWeights::OptimalMIS2Weight( rs.bsdfPdf, envPdf, alpha );
} else {
    w_bsdf = PathTransportUtilities::PowerHeuristic( rs.bsdfPdf, envPdf );
}
```

The PT BSDF-vs-NEE decision is the canonical 2-technique MIS case.
RISE defaults to **power-2** (matches PBRT-v4 / Mitsuba / Cycles).
When `optimalMIS true` is set on the rasterizer, the weight switches
to an alpha-weighted balance heuristic (Kondapaneni et al. SIGGRAPH
2019), where alpha is learned per-pixel from second-moment statistics
during a training phase.  See
[src/Library/Utilities/OptimalMISAccumulator.h](../src/Library/Utilities/OptimalMISAccumulator.h)
and [src/Library/Utilities/MISWeights.h](../src/Library/Utilities/MISWeights.h).

For volumetric scattering — delta tracking vs equiangular sampling —
RISE uses a fixed 50/50 balance:
[RayCaster.cpp:368-388](../src/Library/Rendering/RayCaster.cpp).

### 2. BDPT — power-2

Code: [src/Library/Shaders/BDPTIntegrator.cpp:4378-4575](../src/Library/Shaders/BDPTIntegrator.cpp)
(`MISWeight`).

```cpp
// Strategy (s,t) splits the path into s light verts + t eye verts.
// Walk both subpaths, accumulating ratios r_i = p_i / p_s
// of competing strategies.
sumWeights = 1.0;          // strategy (s,t) itself
for ( each non-delta vertex on light side ) {
    ri *= pdfRev / pdfFwd;
    sumWeights += ri * ri; // power-2
}
for ( each non-delta vertex on eye side ) {
    ri *= pdfRev / pdfFwd;
    sumWeights += ri * ri; // power-2
}
return 1.0 / sumWeights;
```

A compile-time `MISWEIGHT_BALANCE_HEURISTIC` macro at line 4409 swaps
`ri * ri` for `ri` if you ever need to A/B test balance vs power; it's
commented out in production.  Don't enable it in shipped code — it
exists only for diagnostic comparison against SmallVCM-style
references.

The walk uses **`remap0` (Veach/PBRT convention)**: vertices with
`pdfRev = 0` or `pdfFwd = 0` (delta scatters) are mapped to 1 in the
ratio chain so the chain propagates through delta vertices, but they
are then explicitly skipped in the sum.  Without `remap0`, the zero
kills the chain and breaks the weight for any non-delta strategy past
a delta vertex — see the comment at line 4480 for the firefly history.

The `MISWeight` function temporarily clears `isDelta` on the two
**connection** vertices (lines 4448-4464 / 4562-4568).  Connection
evaluation always uses the full non-delta BSDF, regardless of which
lobe was sampled during subpath generation, so for MIS purposes the
connection vertices must be treated as non-delta.  This matches PBRT
convention.

### 3. MLT — inherits BDPT's power-2

`MLTRasterizer` owns a `BDPTIntegrator` and reuses its `MISWeight`.
No MLT-specific MIS.

### 4. VCM — balance (architectural)

Code: [src/Library/Shaders/VCMRecurrence.h:56-59](../src/Library/Shaders/VCMRecurrence.h).

```cpp
inline Scalar VCMMis( const Scalar x )
{
    return x;  // balance heuristic (SmallVCM default)
}
```

VCM cannot use power-2 without a fundamental architectural change.
The Georgiev 2012 paper introduces three running scalars per vertex
(`dVCM`, `dVC`, `dVM`) that let the MIS weight for **every** connection
and merging strategy be evaluated in **O(1) at connection/merge
time** — instead of re-walking every (s,t) pair like BDPT does.
The recursion that maintains those running quantities (see the
`InitLight` / `InitCamera` / `ApplyGeometricUpdate` /
`ApplyBsdfSamplingUpdate` formulas in [VCM.md](VCM.md)) only closes
algebraically at β=1.

The reason is mechanical: each running quantity is a **sum** of
ratios over previous vertices.  The balance heuristic denominator
needs `Σ p_i`, which the running sum computes directly.  The power
heuristic denominator needs `Σ p_i²`, but `(Σ p_i)² ≠ Σ p_i²`, so
maintaining `Σ p_i²` would require a separate running quantity that
tracks **sum of squares of partial pdf products across all
strategies**, plus all the cross-terms from the recurrence
expansion.  At that point you've thrown out the entire reason VCM
beats naive O(s*t) MIS — you might as well use the BDPT walk-based
formula.

There is also a **dynamic-range reason** to prefer balance for VCM:
once the auto-radius shrinks per-sample (SPPM-style progressive
shrinkage, [VCM.md](VCM.md) §"Progressive radius shrinkage"), merging
PDFs vs. connection PDFs differ by ~10 orders of magnitude.  Squaring
those ratios pushes them into FP under/overflow territory and forces
a winner-takes-all between two strategies that **both legitimately
contribute** in caustic regions.  Balance gracefully blends them;
power forces a binary choice between two unbiased estimators.

### 5. SMS — no per-strategy MIS reweight

SMS contributions are splat-accumulated as an auxiliary technique.
The Snell-trace / uniform-area seeding choice and the per-trial
geometric-term clamp (see [CLAUDE.md](../CLAUDE.md) "High-Value
Facts" SMS entries) are upstream of MIS.  SMS doesn't compete in a
multi-technique MIS sum the way BDPT (s,t) strategies do.

## "Why don't they match?  Isn't that a bug?"

This question comes up periodically.  The answer is no, and the
asymmetry is mandated by the math, not chosen by RISE:

1. **BDPT enumerates all (s,t) strategies explicitly.**  There is no
   running quantity to protect; β=2 is "free" and sharpens the
   dominant strategy.
2. **VCM's whole point is the O(1)-per-vertex MIS evaluation via the
   dVCM/dVC/dVM recurrence.**  That recurrence only closes at β=1.
   Switching VCM to β=2 throws out the recurrence and reduces VCM's
   merge-MIS path to BDPT-style explicit enumeration over all
   merging candidates — which would make VCM's per-merge cost scale
   with chain length.

If you genuinely want to revisit this, the principled direction is
**not** to flip VCM to power-2, but to evaluate **optimal MIS
(Kondapaneni 2019)** — RISE already supports it for PT.  Extending
optimal MIS to BDPT is an active research direction (Grittmann et
al. 2019, *Variance-aware MIS*; Grittmann et al. 2021,
*Correlation-aware MIS*); extending to VCM is open research.  The
power-vs-balance debate becomes moot under optimal MIS because the
weight is learned per-pixel from second-moment statistics and
adapts to whichever strategy is locally best.

## When BDPT and VCM disagree on the same scene

On diffuse scenes the two integrators agree to within ~1% at 256 spp.
On caustic-heavy scenes they can differ by more, because BDPT cannot
reach S-D-S-E paths that VCM's merging strategy catches — that's a
**transport-coverage** difference, not an MIS-weighting difference.
If you see VCM and BDPT disagreeing in a region BDPT *should* be able
to reach, walk through
[skills/bdpt-vcm-mis-balance.md](skills/bdpt-vcm-mis-balance.md)
before suspecting the heuristic.
